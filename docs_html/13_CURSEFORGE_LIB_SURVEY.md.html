CurseForge Lib 類別插件調查

本文件調查 CurseForge 上可查到、名稱或用途接近 Lib* 的 WoW 庫類插件，評估其是否適合 EventAlertMod 正式服重寫。

調查原則

  • 未新增硬依賴。
  • 不破壞 EAM 的輕量定位。
  • 不引入 Classic/MOP 相內容層作為正式架構核心。
  • 無需引入高精度掃描、檢查、查詢或大量配置的庫。
  • 若採用，只能走 opt-in 或 soft dependency。
  • 所有庫使用前仍需做Retail 12.x實機載入驗證。

結論摘要

目前最值得保留觀察值的候選只有兩類：

  • LibSharedMedia-3.0：可選，用於字體、音效、材質選單。適合選項階段再評估。
  • LibCustomGlow：任選，用於圖示發光效果。專案已保留舊版參考庫，可優先搶先概念，不建議硬依賴。 其餘庫目前不建議納入 EAM 核心，原因多半是過重、用途不符、更新、依賴太鍊長，或與 EAM 自建的 EventRouter/Scheduler/SavedVariables 方向重疊。

可採用候選選項

LibSharedMedia-3.0

用途：

  • 提供跨 AddOn 的媒體註冊表。
  • 適合日後的UI/Options.lua字型、音效、材質選單。

優點：

  • CurseForge 顯示支援 Retail 12.0.5。
  • 與 EAM 的聲音/font/icon 紋理選項有自然交集。

限制：

  • 不應成為 RequiredDeps。
  • 不應讓渲染器熱路徑依賴它。
  • 只應在選項開啟時查詢媒體清單。

建議整合方式：

OptionalDeps: LibSharedMedia-3.0
if LibStub and LibStub("LibSharedMedia-3.0", true) then
    -- options-only integration
end

目前決策：保留候選，不立即加入。

LibCustomGlow

用途：

  • 圖示發光/像素發光/按鈕發光。

優點：

  • 舊 EAM 已有 LibCustomGlow-1.0 參考資料。
  • 可快速還原「冷卻可用時會高亮」的視覺語意。

限制：

  • 目前專案已移除正式加載的外部依賴。
  • 如果直接加入，會回到舊 LibStub 依賴鏈。
  • 新的渲染器可有效提高材質或暴雪的臨時效果替代。

建議整合方式： - 第一階段：僅參考 ReferenceLibs/LegacyLibs/LibCustomGlow-1.0 的概念。 - 第二階段：若自製成本過高，再討論軟依賴。

目前決策：參考概念，不硬依賴。

明確不採用原因

Ace3 / LibInit

Ace3 是完整的框架，包含生命週期、SavedVariables、事件、配置、GUI、定時器等能力。這些能力與 EAM 正式服重寫的自建核心重疊，而且會增加載入成本。 LibInit 又是 Ace3 封裝層，會硬帶更多模組。 EAM 目前需要的是簡單、可控制、低GC的事件驅動架構，不需要框架化。

LibDataBroker / LibDBIcon

小地圖圖示可以自建。引入 LDB/DBIcon 即可簡單功能變為 LibStub + LDB + DBIcon 依賴鏈，不符合目前「無硬依賴」方向。

LibQTip

EAM 不需要多列工具提示。除錯導出要輸出緊湊的文本，不需要複雜的工具提示框架。

LibRangeCheck / LibGroupInSpecT

這些庫有額外的查詢、檢查或快取行為。 EAM核心不是raid助手，也不需要群組spec/range引擎。

LibDeflate / 序列化類

只有在將來建立大型 import/export 共享格式時才可能需要。 EAM 明確不做 WeakAura 之類的複雜共享系統，因此暫時不採用。

後續建議

短期：

  • 不增加任何函式庫依賴。
  • 渲染器發光先自製最小效果。
  • 選項字體/音效清單先用內建媒體。

中：

  • 若使用者強烈需要跨AddOn媒體選單，再評估LibSharedMedia-3.0軟依賴。
  • 若自製的glow成本過高，再評估LibCustomGlow軟依賴。

長期：

  • 所有第三方函式庫必須投入 ReferenceLibs 先審查。 -經過正式服實機驗證後，才可正式加入TOC。

CurseForge Lib Category Plugin Survey

This document investigates the WoW library class plug-ins that can be found on CurseForge and whose names or uses are close to Lib*, and evaluates whether they are suitable for rewriting in the Retail server of EventAlertMod.

Investigation Principles

  • No new hard dependencies have been added.
  • Don't break lightweight positioning of EAM.
  • Do not introduce Classic/MOP phase content layer as the formal architecture core.
  • No need to introduce high-precision scans, checks, queries, or heavily configured libraries.
  • If adopted, only opt-in or soft dependency can be used.
  • All libraries still need to be verified for Retail 12.x real machine loading before use.

Summary of conclusion

Currently, there are only two categories of candidates most worthy of retaining observations:

  • LibSharedMedia-3.0: Optional, used for font, sound effects, and material menus. Suitable for option phase re-evaluation.
  • LibCustomGlow: optional, used to illustrate the luminous effect. The project has retained the old version of the reference library, and you can give priority to concepts. Hard dependence is not recommended. The remaining libraries are currently not recommended to be included in the EAM core. Most of the reasons are that they are too heavy, have inconsistent uses, are updated, have too long dependencies, or overlap with the self-built EventRouter/Scheduler/SavedVariables direction of EAM.

Alternative options available

LibSharedMedia-3.0

Purpose:

  • Provides media registry across AddOn.
  • Suitable for future UI/Options.lua font, sound effects, and material menus.

Advantages:

  • CurseForge shows support for Retail 12.0.5.
  • Natural intersection with EAM's sound /font/icon texture options.

Limitations:

  • Should not be RequiredDeps.
  • Renderer hot paths should not be made dependent on it.
  • The media list should only be queried when the option is on.

Suggested integration methods:

OptionalDeps: LibSharedMedia-3.0
if LibStub and LibStub("LibSharedMedia-3.0", true) then
    -- options-only integration
end

Current decision: keep the candidate and not join immediately.

LibCustomGlow

Purpose:

  • Icon glow/pixel glow/button glow.

Advantages:

  • Old EAM already has a reference to LibCustomGlow-1.0.
  • The visual semantics of "highlighted when cooldown is available" can be quickly restored.

Limitations:

  • Currently the project has removed officially loaded external dependencies.
  • If added directly, it will return to the old LibStub dependency chain.
  • New renderer that effectively improves materials or Blizzard's temporary effect replacements.

Suggested integration methods: - Phase 1: Only refer to the concept of ReferenceLibs/LegacyLibs/LibCustomGlow-1.0. - Second stage: If the self-made cost is too high, then discuss soft dependencies.

Current decision-making: refer to the concept and do not rely on it.

Clarify the reasons for not adopting

Ace3 / LibInit

Ace3 is a complete framework, including life cycle, SavedVariables, events, configuration, GUI, timer and other capabilities. These capabilities overlap with the EAM self-built core rewritten on the Retail server and will increase the loading cost. LibInit is the Ace3 packaging layer again, which will hard-wire more modules. EAM What is currently needed is a simple, controllable, low GC event-driven architecture that does not require framing.

LibDataBroker / LibDBIcon

Mini map icons can be created by yourself. Introducing LDB/DBIcon can change the simple function to LibStub + LDB + DBIcon dependency chain, which is not in line with the current "no hard dependencies" direction.

LibQTip

EAM Multi-column tooltips are not required. Debug export to output compact text, no need for complex tooltip framework.

LibRangeCheck / LibGroupInSpecT

These libraries have additional querying, checking, or caching behavior. EAM core is not a raid helper and does not require a group spec/range engine.

LibDeflate / Serialization class

This may only be needed if large import/export shared formats are built in the future. EAM explicitly does not build complex sharing systems such as WeakAura, so it is not used for the time being.

Follow-up suggestions

Short term:

  • Does not add any library dependencies.
  • Make the renderer glow first with minimum effect.

  • Options font/sound effects list uses built-in media first. Medium:

  • If the user strongly needs to cross the AddOn media menu, then evaluate the LibSharedMedia-3.0 soft dependency.

  • If the cost of self-made glow is too high, re-evaluate the LibCustomGlow soft dependency.

Long term:

  • All third-party libraries must be reviewed first by setting ReferenceLibs. -You can officially join TOC only after passing the Retail server verification.
