CurseForge Lib 類插件調查

本文件調查 CurseForge 上可查到、名稱或用途接近 Lib* 的 WoW library 類插件，評估其是否適合 EventAlertMod Retail rewrite。

調查原則

  • 不新增硬依賴。
  • 不破壞 EAM 的輕量定位。
  • 不引入 Classic/MOP 相容層作為正式架構核心。
  • 不引入會高頻掃描、inspect、query 或大量配置的 library。
  • 若採用，只能走 opt-in 或 soft dependency。
  • 所有 library 使用前仍需做 Retail 12.x 實機載入驗證。

結論摘要

目前最值得保留觀察的候選只有兩類：

  • LibSharedMedia-3.0：可選，用於字型、音效、材質選單。適合 options 階段再評估。
  • LibCustomGlow：可選，用於 icon glow 效果。專案已保留舊版參考庫，可優先借鑑概念，不建議硬依賴。

其餘 library 目前不建議納入 EAM 核心，原因多半是過重、用途不符、更新停滯、依賴鏈太長，或與 EAM 自建的 EventRouter/Scheduler/SavedVariables 方向重疊。

可採用候選細節

LibSharedMedia-3.0

用途：

  • 提供跨 AddOn 的 media registry。
  • 適合日後 UI/Options.lua 的字型、音效、材質選單。

優點：

  • CurseForge 顯示支援 Retail 12.0.5。
  • 與 EAM 的 sound/font/icon texture options 有自然交集。

限制：

  • 不應成為 RequiredDeps。
  • 不應讓 Renderer hot path 依賴它。
  • 只應在 Options 開啟時查詢 media list。

建議整合方式：

OptionalDeps: LibSharedMedia-3.0
if LibStub and LibStub("LibSharedMedia-3.0", true) then
    -- options-only integration
end

目前決策：保留候選，不立即加入。

LibCustomGlow

用途：

  • icon glow / pixel glow / button glow。

優點：

  • 舊 EAM 已有 LibCustomGlow-1.0 參考資料。
  • 可快速還原「冷卻可用時高亮」的視覺語意。

限制：

  • 目前專案已移除正式載入的 external dependency。
  • 若直接加入，會回到舊 LibStub 依賴鏈。
  • 新 Renderer 可用簡化材質或 Blizzard 原生效果替代。

建議整合方式：

  • 第一階段：只參考 ReferenceLibs/LegacyLibs/LibCustomGlow-1.0 的概念。
  • 第二階段：若自製 glow 成本過高，再討論 soft dependency。

目前決策：參考概念，不硬依賴。

明確不採用原因

Ace3 / LibInit

Ace3 是完整 framework，包含 lifecycle、SavedVariables、event、config、GUI、timer 等能力。這些能力與 EAM Retail rewrite 的自建核心重疊，而且會增加載入成本。LibInit 又是 Ace3 包裝層，會硬帶更多模組。

EAM 目前需要的是簡單、可控、低 GC 的 event-driven 架構，不需要 framework 化。

LibDataBroker / LibDBIcon

Minimap icon 可以自建。引入 LDB/DBIcon 會把簡單功能變成 LibStub + LDB + DBIcon 依賴鏈，不符合目前「無硬依賴」方向。

LibQTip

EAM 不需要 multi-column tooltip。Debug export 應輸出 compact text，不需要複雜 tooltip frame。

LibRangeCheck / LibGroupInSpecT

這些 library 都有額外查詢、inspect 或 cache 行為。EAM 核心不是 raid assistant，也不需要 group spec/range engine。

LibDeflate / Serialize 類

只有在將來建立大型 import/export 分享格式時才可能需要。EAM 明確不做 WeakAura-like 複雜分享系統，因此暫不採用。

後續建議

短期：

  • 不新增任何 Lib 依賴。
  • Renderer glow 先自製最小效果。
  • Options 字型/音效清單先用內建 media。

中期：

  • 若使用者強烈需要跨 AddOn media 選單，再評估 LibSharedMedia-3.0 soft dependency。
  • 若自製 glow 成本過高，再評估 LibCustomGlow soft dependency。

長期：

  • 所有第三方 library 必須放入 ReferenceLibs 先審查。
  • 經過 Retail 實機驗證後，才可加入正式 TOC。

CurseForge Lib class plug-in survey

This document investigates WoW library plugins with names or uses close to Lib* that can be found on CurseForge and evaluates whether they are suitable for EventAlertMod Retail rewrite.

Investigation Principles

  • No new hard dependencies are added.
  • Lightweight positioning without breaking EAM.
  • No Classic/MOP compatibility layer is introduced as a formal architectural core.
  • Do not introduce libraries that require high-frequency scanning, inspect, query, or extensive configuration.
  • If adopted, only opt-in or soft dependency can be used.
  • All libraries still require Retail 12.x real-machine loading verification before use.

Conclusion summary

Currently, there are only two categories of candidates most worthy of observation:

  • LibSharedMedia-3.0: Optional, used for font, sound effects, and material menus. Suitable for re-evaluation in options phase.
  • LibCustomGlow: optional, used for icon glow effect. The project has retained the old version of the reference library, and you can give priority to learning concepts. Hard dependence is not recommended. The remaining libraries are currently not recommended to be included in the EAM core. Most of the reasons are that they are too heavy, do not match the purpose, update is stagnant, the dependency chain is too long, or overlap with EAM's self-built EventRouter/Scheduler/SavedVariables direction.

Candidate details can be adopted

LibSharedMedia-3.0

Purpose:

  • Provide media registry across AddOn.
  • Suitable for future UI/Options.lua font, sound effects, and material menus.

Advantages:

  • CurseForge displays support for Retail 12.0.5.
  • Natural intersection with EAM's sound/font/icon texture options.

Limitations:

  • Should not be RequiredDeps.
  • Renderer hot path should not be made dependent on it.
  • The media list should only be queried when Options is on.

Suggested integration methods:

OptionalDeps: LibSharedMedia-3.0
if LibStub and LibStub("LibSharedMedia-3.0", true) then
    -- options-only integration
end

Current decision: keep the candidate and not join immediately.

LibCustomGlow

Purpose:

  • icon glow / pixel glow / button glow.

Advantages:

  • Old EAM already has LibCustomGlow-1.0 reference.
  • The visual semantics of "highlight when cooldown is available" can be quickly restored.

Limitations:

  • The currently loaded external dependency has been removed from the project.
  • If added directly, it will return to the old LibStub dependency chain.
  • New Renderer can be replaced with simplified materials or Blizzard native effects.

Suggested integration methods: - Phase 1: Only refer to the concepts of ReferenceLibs/LegacyLibs/LibCustomGlow-1.0. - Second stage: If the cost of self-made glow is too high, then discuss soft dependency.

Current decision-making: refer to the concept and do not rely on it.

Clarify the reasons for not adopting

Ace3/LibInit

Ace3 is a complete framework, including lifecycle, SavedVariables, event, config, GUI, timer and other capabilities. These capabilities overlap with EAM Retail rewrite's self-built core and will increase onboarding costs. LibInit is another Ace3 packaging layer that will hard-wire more modules. What EAM currently needs is a simple, controllable, low GC event-driven architecture that does not require frameworkization.

LibDataBroker/LibDBIcon

Minimap icon can be built by yourself. Introducing LDB/DBIcon will turn simple functions into a LibStub + LDB + DBIcon dependency chain, which is not in line with the current "no hard dependencies" direction.

LibQTip

EAM does not require multi-column tooltips. Debug export should output compact text, no complex tooltip frame is required.

LibRangeCheck / LibGroupInSpecT

These libraries have additional query, inspect or cache behavior. The EAM core is not a raid assistant and does not require a group spec/range engine.

LibDeflate / Serialize class

This may only be needed if large import/export sharing formats are built in the future. EAM clearly does not build a WeakAura-like complex sharing system, so it will not be used for the time being.

Follow-up suggestions

Short term:

  • No new Lib dependencies are added.
  • Renderer glow Make your own minimum effect first.
  • Options font/sound effects list uses built-in media first.

Mid-term:

  • If users strongly need to cross the AddOn media menu, re-evaluate the LibSharedMedia-3.0 soft dependency.
  • If the cost of homemade glow is too high, re-evaluate the LibCustomGlow soft dependency.

Long term:

  • All third-party libraries must be put into ReferenceLibs and reviewed first.
  • The official TOC can be added only after it has been verified by Retail's actual machine.
