Performance & Taint Control 審查報告：統一 OnUpdate 計時器與 Layout 偏移陣列

[!NOTE] 本文件由 EAM Performance & Taint Control Expert 撰寫，旨在評估 EventAlertMod 重構版中統一計時器（legacyTimer）的運作開銷、LAYOUT_OFFSETS 靜態偏移陣列的算術性能，並識別戰鬥中潛在的隱性 Taint 洩漏與 GC 垃圾分配問題。

1. 統一 OnUpdate 計時器（legacyTimer）運作開銷審查

UI/Renderer.lua 中的 legacyTimer 負責在不使用原生 Duration Binding 時，提供備用的秒數倒數顯示（尤其在 $timeLeft < 3.05$ 秒時提供小數點後一位倒數）。其運作機制為使用一個唯一的匿名偵聽 Frame 監聽 OnUpdate，遍歷 activeLegacyTimers 與 activeDurationObjects 以更新圖示的 timerText。

1.1 性能優勢

  • 單一 Frame 集中驅動：避免了每個圖示或每個法術各自綁定 OnUpdate 的高開銷設計。當沒有 active 任務時，會調用 legacyTimerFrame:SetScript("OnUpdate", nil) 解除綁定，保持 CPU 零占用。
  • 孤兒 Frame 隔離：legacyTimerFrame 為匿名無 Parent 的 Frame（CreateFrame("Frame")），完全與 UIParent 核心鏈解耦，消除了事件派發過程中的 Taint 傳播風險。

1.2 運作開銷與 GC 瓶頸分析

在高頻戰鬥環境（例如 60-120 FPS，擁有多個活動監控圖示）下，目前的實作存在顯著的 GC 與 CPU 開銷： 1. 頻繁的字串格式化 (GC Heap 垃圾主要來源)： * 在 onLegacyTimerUpdate 中，每幀對所有 active icon 執行： lua if timeLeft < 3.05 then text = string.format("%.1f", timeLeft) else text = string.format("%d", math.ceil(timeLeft)) end * 在 Lua VM 中，string.format 會產生全新的字串對象。在 60 FPS 下，若有 5 個圖示同時倒數，一秒將配置 $60 \times 5 = 300$ 個字串。這會直接加速 GC 觸發，造成幀率抖動（Micro-stutter）。 2. pairs 雜湊表迭代開銷： * 使用 pairs(activeLegacyTimers) 來遍歷活動計時器。在 LuaJIT 中，若遍歷雜湊表（Hash Part），JIT 編譯器可能無法有效進行 Trace Compiler 編譯，且迭代過程中對 table 進行 activeLegacyTimers[icon] = nil 的刪除操作，容易引起內部 Hash 結構重整與 GC 開銷。 3. pcall 所致的 JIT Trace Abort： * 在 activeDurationObjects 迭代中，調用了 safeCheckIsZero(durationObj)，其內部包含 pcall(durationObj.IsZero, durationObj)。 * 在 LuaJIT 中，pcall 會直接導致當前的 Trace 編譯失敗（Trace Abort），使得 OnUpdate 的核心執行路徑只能退回到低效的 interpreter 模式執行，大幅增加了每幀的 CPU 算術與調用開銷。

2. 7 大 Alert Frame Layout 靜態連續偏移陣列（LAYOUT_OFFSETS）審查

EAM 定義了 7 大獨立告警框架，並在排版時利用 EAM.Constants.LAYOUT_OFFSETS 做為位移向量：

LAYOUT_OFFSETS = freeze({
    freeze({ 1, 0 }),  -- 1 = RIGHT (向右成長)
    freeze({ -1, 0 }), -- 2 = LEFT (向左成長)
    freeze({ 0, 1 }),  -- 3 = UP (向上成長)
    freeze({ 0, -1 })  -- 4 = DOWN (向下成長)
})

2.1 CPU 算術開銷評估

在 layout(frameName) 執行時，其定位計算為：

local offset = EAM.Constants.LAYOUT_OFFSETS[dirIdx]
local dx, dy = offset[1], offset[2]
...
local dist = (layoutIndex - 1) * (size + spacing)
local offsetX = dx * dist
local offsetY = dy * dist
  • 算術性能極佳：相較於舊版 EventAlert 的多重 if-else 或 switch 分支判斷（如 if dir == 1 then x = ... else ...），以連續數字索引陣列配合乘法運算，完全消除了 CPU 分支預測錯誤（Branch Misprediction）的可能。
  • 唯讀凍結保障：使用了 table.freeze 確保該常數陣列在執行期不可被篡改，兼顧安全與效能。
  • 查表開銷評估：雖然每次定位都需要從 LAYOUT_OFFSETS 讀取內部 table 元素，但在實際運作中，layout 僅在圖示增減或設定變更時被觸發（由 BeginBatch/EndBatch 節流，並非高頻熱路徑），因此其產生的微小查表開銷在整體 AddOn 中微不足道，屬於極佳的設計決策。

3. 戰鬥中隱性 Taint 洩漏與 GC heap 垃圾分配評估

經深入代碼走讀，我們發現了以下在戰鬥中高頻事件與渲染下，可能導致隱性 Taint 洩漏或 GC heap 垃圾分配的關鍵問題：

3.1 隱性 Taint 洩漏與戰鬥鎖定 (Combat Lockdown) 隱患

  1. 戰鬥中動態 CreateFrame (高風險點)：
    • 雖然 ensureParent 具備 InCombatLockdown 守衛，但在 Renderer.render 中，如果 IconPool 的可用緩衝圖示用盡（例如在戰鬥中同時觸發多個天賦/冷卻/法術，使得 active 數量大於 prewarm 的 16 個），會觸發 IconPool.acquire() -> createIcon()： lua local function createIcon() local button = api.CreateFrame("Frame", name, UIParent) ... local cooldown = api.CreateFrame("Cooldown", nil, button, "CooldownFrameTemplate")
    • 風險：在戰鬥鎖定期間呼叫 CreateFrame("Frame", name, UIParent) 且載入了官方的 CooldownFrameTemplate，可能會污染暴雪內部的 Secure Template Chain，並高機率觸發 "Action blocked by an addon" 錯誤。
  2. 影子載體 (Shadow Host) 寄生模式的潛在 Taint 傳播 (極高風險點)：
    • 雖然目前 useCDM 被強制設為 false，但若啟用此模式，代碼會將 EAM 的 icon 寄生在官方原生 CooldownViewer 或 ActionButton 下： lua icon:SetParent(hostIcon) icon:ClearAllPoints() icon:SetAllPoints(hostIcon)
    • 風險：官方的 CooldownViewer 與 ActionButton 屬於高度受保護的 Secure / Protected Frame。非安全的 EAM 圖示在戰鬥中呼叫 SetParent 附加到 Secure Frame 上，當暴雪 UI 試圖重新排版或操作該 Secure Frame 時，會因為其子系包含 Unsecure Frame 而導致整條執行鏈被標記為 Taint，進而引發戰鬥中 Action Button 被阻擋、無法點擊的致命錯誤。

3.2 隱性 GC Heap 垃圾分配隱患

  1. 熱路徑中的匿名閉包 pcall：
    • 在 Renderer.render 中： lua pcall(function() icon:SetFrameStrata("MEDIUM") icon:SetFrameLevel(hostIcon:GetFrameLevel() + 10) end)
    • 每次渲染更新（如堆疊數、冷卻更新）時，都會在熱路徑中動態分配這兩個匿名 closure 函數，並調用 pcall。這不僅會阻礙 LuaJIT VM 的 Trace Compilation，還會產生持續的 GC 垃圾堆積。
    • 註：icon.overlay 與 icon.cooldown 皆為 EAM 內部建立的 Frame，其方法是 100% 確定存在的，根本不需要使用 pcall 進行防護。

4. 具體重構與優化建議

為了消除上述隱患並達成 EAM 重寫版的極致性能與零 Taint 要求，建議實施以下優化重構：

4.1 消除字串分配與 pairs 迭代：引進「時間字串快取」與「雙向鏈結串列」

  • 時間字串快取 (Time Text Cache)： 預先生成 $0.0$ 到 $3.0$ 秒（間隔 0.1 秒，共 30 個）以及 $3$ 到 $3600$ 秒的整數字串表。在 OnUpdate 中，通過數值計算直接索引此快取表，將格式化開銷降為零。
  • 平坦雙向鏈結/數值索引陣列： 將 activeLegacyTimers 改為使用數值索引的平坦陣列，藉此使用 for i = 1, count 進行數字循環迭代，消除 pairs 在高頻路徑下的 iterator 分配與雜湊查表開銷。

4.2 移除熱路徑 pcall 與匿名閉包

  • 移除 Renderer.render 中對自建框架的 pcall(function() ... end) 包裝，改為直接賦值或使用預先定義好的靜態輔助函數，避免高頻分配。

4.3 強制執行戰鬥中 CreateFrame 與 SetParent 延遲

  • 在 IconPool.acquire() 中加入 InCombatLockdown 檢測。若在戰鬥中且無可用 icon，應降級或將其加入延遲渲染佇列（deferRender），嚴禁在戰鬥鎖定期間呼叫 api.CreateFrame。
  • 徹底廢除或嚴格限制影子載體（Shadow Host）寄生模式在戰鬥中的 SetParent 行為，避免任何與 Secure Frame 的直接 parent 連結。

  • 建議決策：在下一個重構週期中，應立即針對 UI/Renderer.lua 與 UI/IconPool.lua 進行局部重構，落實「時間字串快取」、「無 pcall 直接調用」以及「戰鬥中 CreateFrame 延遲守衛」，以確保 AddOn 運作時的極致低 GC 與絕對安全。

Performance & Taint Control Review Report: Unifying OnUpdate Timer and Layout Offset Array

[!NOTE] This document is written by EAM Performance & Taint Control Expert to evaluate the operational overhead of the legacy timer (legacyTimer) in the refactored version of EventAlertMod, the arithmetic performance of the LAYOUT_OFFSETS static offset array, and to identify potential hidden Taint leaks and GC garbage allocation issues in combat.

1. Unified OnUpdate timer (legacyTimer) operation overhead review

legacyTimer in UI/Renderer.lua is responsible for providing an alternative countdown display of seconds when the native Duration Binding is not used (especially providing one decimal place countdown when $timeLeft < 3.05$ seconds). Its operating mechanism is to use a unique anonymous listening Frame to listen to OnUpdate, and traverse activeLegacyTimers and activeDurationObjects to update the timerText shown in the diagram.

1.1 Performance advantages

  • Single Frame centralized driver: avoids the high overhead design of binding OnUpdate to each icon or each spell. When there is no active task, legacyTimerFrame:SetScript("OnUpdate", nil) will be called to unbind and maintain zero CPU usage.
  • Orphan Frame Isolation: legacyTimerFrame is an anonymous Parentless Frame (CreateFrame("Frame")), completely decoupled from the UIParent core chain, eliminating the risk of Taint propagation in the event dispatch process.

1.2 Operational overhead and GC bottleneck analysis

In high-frequency combat environments (e.g. 60-120 FPS, with multiple active monitor graphs), the current implementation has significant GC and CPU overhead: 1. Frequent string formatting (main source of GC Heap garbage): * In onLegacyTimerUpdate, execute for all active icons every frame: lua if timeLeft < 3.05 then text = string.format("%.1f", timeLeft) else text = string.format("%d", math.ceil(timeLeft)) end * In Lua VM, string.format will generate a brand new string object. At 60 FPS, if there are 5 icons counting down at the same time, $60 \times 5 = 300$ strings will be configured in one second. This will directly accelerate GC triggering and cause frame rate jitter (Micro-stutter). 2. pairs hash table iteration overhead: * Use pairs(activeLegacyTimers) to iterate over active timers. In LuaJIT, if the hash table (Hash Part) is traversed, the JIT compiler may not be able to effectively perform Trace Compiler compilation, and the deletion operation of activeLegacyTimers[icon] = nil on the table during the iteration process may easily cause internal Hash structure reorganization and GC overhead. 3. pcall caused JIT Trace Abort: * In the activeDurationObjects iteration, safeCheckIsZero(durationObj) is called, which contains pcall(durationObj.IsZero, durationObj) inside. * In LuaJIT, pcall will directly cause the current Trace compilation to fail (Trace Abort), so that the core execution path of OnUpdate can only fall back to the inefficient interpreter mode execution, which greatly increases the CPU arithmetic and calling overhead of each frame.

2. 7 Big Alert Frame Layout Static Continuous Offset Arrays (LAYOUT_OFFSETS) Review

EAM defines 7 independent alarm frames and uses EAM.Constants.LAYOUT_OFFSETS as the displacement vector during typesetting:

LAYOUT_OFFSETS = freeze({
    freeze({ 1, 0 }),  -- 1 = RIGHT (向右成長)
    freeze({ -1, 0 }), -- 2 = LEFT (向左成長)
    freeze({ 0, 1 }),  -- 3 = UP (向上成長)
    freeze({ 0, -1 })  -- 4 = DOWN (向下成長)
})

2.1 CPU arithmetic overhead evaluation

When layout(frameName) is executed, its positioning is calculated as:

local offset = EAM.Constants.LAYOUT_OFFSETS[dirIdx]
local dx, dy = offset[1], offset[2]
...
local dist = (layoutIndex - 1) * (size + spacing)
local offsetX = dx * dist
local offsetY = dy * dist
  • Excellent arithmetic performance: Compared with the multiple if-else or switch branch judgments of the old version of EventAlert (such as if dir == 1 then x = ... else ...), the use of continuous digital index arrays with multiplication operations completely eliminates the possibility of CPU branch misprediction (Branch Misprediction).
  • Read-only freeze guarantee: table.freeze is used to ensure that the constant array cannot be tampered with during execution, taking into account both security and performance.
  • Table lookup cost evaluation: Although each positioning requires reading internal table elements from LAYOUT_OFFSETS, in actual operation, layout is only triggered when icons are added or removed or settings are changed (throttled by BeginBatch/EndBatch, not a high-frequency hot path), so the tiny table lookup overhead it generates is negligible in the overall AddOn, and is an excellent design decision.

3. Implicit Taint leaks and GC heap garbage allocation evaluation in combat

After in-depth code reading, we discovered the following key issues that may lead to hidden Taint leaks or GC heap garbage allocation under high-frequency events and rendering in combat:

3.1 Hidden Taint Leakage and Combat Lockdown Hazards

  1. Dynamic CreateFrame during battle (high risk point):
  2. Although ensureParent has the InCombatLockdown guard, in Renderer.render, if the available buffer icons of IconPool are exhausted (for example, multiple talents/cooldowns/spells are triggered at the same time in combat, making the active number greater than the 16 in prewarm), IconPool.acquire() -> createIcon() will be triggered: lua local function createIcon() local button = api.CreateFrame("Frame", name, UIParent) ... local cooldown = api.CreateFrame("Cooldown", nil, button, "CooldownFrameTemplate")
  3. Risk: Calling CreateFrame("Frame", name, UIParent) during combat lock and loading the official CooldownFrameTemplate may pollute Blizzard's internal Secure Template Chain and trigger an "Action blocked by an addon" error with a high probability.
  4. Potential Taint propagation of Shadow Host parasitic mode (extremely high risk point):
  5. Although useCDM is currently forced to false, if this mode is enabled, the code will parasitize the EAM icon under the official native CooldownViewer or ActionButton: lua icon:SetParent(hostIcon) icon:ClearAllPoints() icon:SetAllPoints(hostIcon)
  6. Risk: The official CooldownViewer and ActionButton belong to the highly protected Secure / Protected Frame. The unsafe EAM icon calls SetParent and is attached to the Secure Frame during combat. When Blizzard UI attempts to rearrange or operate the Secure Frame, the entire execution chain will be marked as Taint because its subsystem contains Unsecure Frame, which will cause a fatal error in which the Action Button is blocked and cannot be clicked during combat.

3.2 Hidden GC Heap garbage allocation hazards

  1. Anonymous closure pcall in hot path:
    • In Renderer.render: lua pcall(function() icon:SetFrameStrata("MEDIUM") icon:SetFrameLevel(hostIcon:GetFrameLevel() + 10) end)
  2. Each time the rendering is updated (such as stack count, cooling update), these two anonymous closure functions are dynamically allocated in the hot path and pcall is called. Not only does this hinder Trace Compilation of the LuaJIT VM, but it also creates ongoing GC garbage accumulation.
    • *Note: icon.overlay and icon.cooldown are both Frames created internally by EAM. Their methods are 100% certain to exist and there is no need to use pcall for protection. *

4. Specific reconstruction and optimization suggestions

In order to eliminate the above hidden dangers and achieve the ultimate performance and zero Taint requirements of the EAM rewritten version, it is recommended to implement the following optimization and reconstruction:

4.1 Eliminate string allocation and pairs iteration: introduce "time string cache" and "bidirectional chained list"

  • Time Text Cache: Pre-generate a list of integer strings from $0.0$ to $3.0$ seconds (0.1 second intervals, 30 in total) and $3$ to $3600$ seconds. In OnUpdate, this cache table is indexed directly through numerical calculations, reducing the formatting overhead to zero.
  • Flat bidirectional link/numeric index array: Change activeLegacyTimers to use a numerically indexed flat array, thereby using for i = 1, count for numeric loop iteration, eliminating the iterator allocation and hash table lookup overhead of pairs in high-frequency paths.

4.2 Remove hot path pcall and anonymous closure

  • Remove the pcall(function() ... end) wrapper for self-built frameworks in Renderer.render and replace it with direct assignment or use predefined static auxiliary functions to avoid high-frequency allocation.

4.3 Force CreateFrame and SetParent delays in execution combat

  • Add InCombatLockdown detection to IconPool.acquire(). If you are in combat and there is no available icon, you should downgrade it or add it to the deferred rendering queue (deferRender). Calling api.CreateFrame during combat lock is strictly prohibited.
  • Completely abolish or strictly limit the SetParent behavior of Shadow Host parasitic mode in combat to avoid any direct parent connection with Secure Frame.
