EventAlertMod 品質控制與要因分析指引（QC 與根本原因分析指引）

少年欸！為確保EAM在重構與日後開發中的程序代碼質量，避免在同一個Bug上反覆栽種，本文件正式將企業管理學的控製品質（TQM）、新QC七大手段、5個為什麼（WHY-WHYDE）、魚圖要因分析、矩陣對策評估與甘特，圖時相分析的精神法。 後續所有子代理進行問題排查、方案設計（實施方案）與PR審查，均須遵循此QC指引。

1. 5 Whys (WHY-WHY要因分析) 實施規範

當 EAM 出現任何 Bug、Taint 爆發或 Lua 報錯時，主代理與子代理不得直接嘗試修改程序碼，必須先在 Docs/15_DEVELOPMENT_ISSUE_LOG.md 記錄中進行 5 個為什麼分析，連續追問 5 次「為什麼」，直到根本原因（Root Cause）。

範例：戰鬥中 IsZero() 布林危機崩潰分析

  • 為什麼1（一問）：為什麼OnUpdate執行時外掛程式並崩潰UI？
    • 答：因為在Renderer.lua:97對durationObj:IsZero()的傳回值進行了直接布林判斷（if durationObj:IsZero() then）。
  • 為什麼2（二問）：為什麼 IsZero() 會引發崩潰？
  • 答：因為該 DurationObject 代表的光環在戰鬥中返回保密限制，其IsZero()方法是一個受保護的Secret Boolean。
  • 為什麼3（三問）：為什麼受保護的秘密布爾不能做布林判斷？
    • 答：因為魔獸世界 12.x 引擎為了阻止 AddOn 水晶由 API 判定來輔助戰鬥決策，嚴禁在 Lua 中對 Secret Boolean 執行布林芥末指令（Boolean Test）。
  • 為什麼4（四問）：為什麼我們非要在OnUpdate中呼叫IsZero()？
    • 回答：因為戰鬥中UNIT_AURA事件會被官方節流，且秘密狀態下我們無法取得剩餘時間以預排調度器，必須在渲染器輪詢該謂詞以便在光環消失時主動恢復圖示。
  • 為什麼5（五問/根本原因）：為什麼之前沒有防護？
  • 答：因為渲染器缺乏對Secret Value的前置檢測與安全包裹，且不知道IsZero()的傳回值和是Secret。
  • 根本對策：引入safeCheckIsZero，利用Util.isSecretValue過濾回傳值，若為Secret則本身進行布林測試。

2. 魚骨圖 (Ishikawa Diagram) 要因分析模板

在遇到複雜的系統性問題時（例如「一個 ICON 都沒有出現」或「戰鬥高層中 UI 嚴重卡頓」），必須使用魚骨圖從以下五個維度進行分析：

  【人 (Agent)】                   【機 (Client/WoW API)】         【料 (SavedVariables/Data)】
  - 載入順序快取 nil 漏配           - 12.x Secrecy 加密與限制       - 舊 SavedVariables 資料遷移失效
  - 靜態變數被提早快取             - 事件節流與 CLEU 事件停用       - 預設 Alerts 表為空 (WTF遺失)
          \                             \                               \
           \                             \                               \
            --------------------------------------------------------------------> [ 系統異常/Bug ]
           /                             /                               /
          /                             /                               /
  - 戰鬥中 layout 頻繁重排         - 戰鬥中 InCombatLockdown 限制
  - 輪詢 pcall 產生 Heap 垃圾       - 脫戰後 defer 釋放機制
  【法 (Algorithm/邏輯)】          【環 (Environment/戰鬥狀態)】
  • 人（代理/開發人員）：AI代理的程式碼載入順序、快速取邏輯、靜態變數使用不當。
  • ** 機器 (Client / WoW API)**：魔獸客戶端 API 的行為變更、加密限制、事件節流。
  • 資料 (SavedVariables / Data)：資料庫架構、預設設定檔、在地化字串。
  • 法（演算法/邏輯流程）：演算法費用、迴圈設計、事件註冊、GC垃圾產生率。
  • 環（環境/遊戲狀態）：戰鬥中/非戰鬥中、低FPS、大量Buff特殊恐懼。

4.甘特圖排程與依賴精神（Gantt Spirit）

在 task.md 中，必須反映甘特圖的依賴關係（Dependency）與關鍵路徑（Critical Path），以防止工具開發的子代理發生「因前置方案碼未就緒而空轉」或「高關聯方案碼衝突」。

程排原則：

  1. Dependency Guard (依賴)：在 task.md 中明確標誌依賴關係。例如：
    • [ ] 任務 C (AuraService 重構) [依賴：任務 A (EventRouter 改版)]
  2. 關鍵路徑優先（關鍵路徑優先）：將決定架構穩定性（如Core, SavedVariables）的任務排在最前面，與一級審查專家分配最昂貴的資源。
  3. Milestone Gate (里程碑關卡)：每個階段結束時，必須設定 Milestone 進行品質查核（如：luac -p檢查，/eam doctor診斷通過），否則無法進入階段。

5. 新的 QC 七大手段在 EAM 的落地

  1. 關係圖法（RelationsDiagram）：
    • 應用：釐清EAM內部事件（如EAM_AURA_STATE_CHANGED）與WoW事件（如UNIT_AURA）之間的因果傳播網絡，防止事件風暴與無限傳回。
  2. 系統圖法（Systematic Chart）：
    • 應用：將巨集大的重構目標（如「12.1.0 零分配」）分層分層為子模組變更（AuraService池化、CooldownService池化、AlertManager重複、渲染器多型渲染）。
  3. ** 矩陣圖法（Matrix Diagram）**：
    • 應用：即本案已落地的Docs/21_RACI_EXPERTS_MATRIX.md，用於釐清專家與開發任務的責任對照。
  4. PDPC法 (Process Decision Program Chart / 流程決定計畫畫圖)：
  5. 應用：在程式碼編寫前，預測所有可能的異常路徑（如：官方API返回nil、資料未快取、戰鬥中設定限制），並在程式碼中預先準備pcall故障隔離、safeCheckIsZero和Util.isSecretValue等後備備用對策。
  6. 箭條圖法（箭頭圖）：
  7. 應用：理清子代理的工件工作順序，避免多個代理同時寫入同一個文件，確保版本分支的平滑合併。

EventAlertMod Quality Control and Root Cause Analysis Guidelines (QC and Root Cause Analysis Guidelines)

Boy! In order to ensure the program code quality of EAM during reconstruction and future development and avoid planting the same bug repeatedly, this document formally combines the control quality of enterprise management (TQM), the seven new QC methods, the 5 whys (WHY-WHYDE), fish chart factor analysis, matrix strategy evaluation and Gantt, the mental method of chart phase analysis. All subsequent sub-agents must follow this QC guideline when conducting problem troubleshooting, plan design (implementation plan) and PR review.

1. 5 Whys (WHY-WHY factor analysis) implementation specifications

When any bug, Taint outbreak or Lua error occurs in EAM, the main agent and sub-agent must not directly try to modify the program code. They must first perform 5 why analysis in the Docs/15_DEVELOPMENT_ISSUE_LOG.md record and ask "why" 5 times in a row until the root cause (Root Cause) is found.

Example: IsZero() Bryn Crisis Collapse Analysis in Combat

  • Why 1 (one question): Why does the plug-in program and crash the UI when OnUpdate is executed?
    • Answer: Because in Renderer.lua:97 a direct Boolean judgment is made on the return value of durationObj:IsZero() (if durationObj:IsZero() then).
  • Why 2 (second question): Why does IsZero() cause a crash?
  • Answer: Because the DurationObject represents the confidentiality limit returned by the Aura in combat, its IsZero()method is a protectedSecret Boolean`.
  • Why 3 (Three Questions): Why can’t the protected Secret Boolean do Bollinger judgment?
    • Answer: Because the World of Warcraft 12.x engine prevents AddOn crystals from being determined by API to assist combat decision-making, it is strictly prohibited to execute Boolean Test instructions on Secret Boolean in Lua.
  • Why 4 (four questions): Why do we have to call IsZero() in OnUpdate?
    • Answer: Because the UNIT_AURA event in combat will be officially throttled, and in the secret state we cannot obtain the remaining time to pre-schedule the scheduler, the renderer must poll this predicate to actively restore the icon when the Aura disappears.
  • Why 5 (Five Questions/Root Reasons): Why was there no protection before?
  • Answer: Because the renderer lacks pre-detection and security wrapping of Secret Value, and does not know that the return value of IsZero() is a Secret.
  • Basic Countermeasure: Introduce safeCheckIsZero, use Util.isSecretValue to filter the returned value, and if it is a Secret, perform the Boolean test yourself.

2. Fishbone Diagram (Ishikawa Diagram) factor analysis template

When encountering complex systemic problems (such as "not a single ICON appears" or "the UI is severely stuck in the high-level combat"), a fishbone diagram must be used to analyze from the following five dimensions:

  【人 (Agent)】                   【機 (Client/WoW API)】         【料 (SavedVariables/Data)】
  - 載入順序快取 nil 漏配           - 12.x Secrecy 加密與限制       - 舊 SavedVariables 資料遷移失效
  - 靜態變數被提早快取             - 事件節流與 CLEU 事件停用       - 預設 Alerts 表為空 (WTF遺失)
          \                             \                               \
           \                             \                               \
            --------------------------------------------------------------------> [ 系統異常/Bug ]
           /                             /                               /
          /                             /                               /
  - 戰鬥中 layout 頻繁重排         - 戰鬥中 InCombatLockdown 限制
  - 輪詢 pcall 產生 Heap 垃圾       - 脫戰後 defer 釋放機制
  【法 (Algorithm/邏輯)】          【環 (Environment/戰鬥狀態)】
  • People (Agent/Developer): AI agent's code loading sequence, quick fetch logic, and improper use of static variables.
  • ** Machine (Client / WoW API)**: Behavior changes, encryption restrictions, event throttling for Warcraft client API.
  • Data (SavedVariables / Data): database structure, default profile, localization strings.
  • Method (algorithm/logical process): algorithm cost, loop design, event registration, GC garbage generation rate.
  • Ring (Environment/Game State): In combat/not in combat, low FPS, a lot of Buff special fears.

4. Gantt Chart Scheduling and Dependency Spirit (Gantt Spirit)

In task.md, the Dependency and Critical Path of the Gantt chart must be reflected to prevent the sub-agent developed by the tool from "idling due to unready pre-program code" or "high correlation program code conflict".

Scheduling principles:

  1. Dependency Guard (dependency): Clearly mark dependencies in task.md. For example:
    • [ ] Task C (AuraService Refactored) [Depends on: Task A (EventRouter Revised)]
  2. Critical Path First (Critical Path First): Prioritize tasks that determine the stability of the architecture (such as Core, SavedVariables), and allocate the most expensive resources with first-level review experts.
  3. Milestone Gate (milestone level): At the end of each stage, Milestone must be set for quality inspection (such as: luac -p check, /eam doctor diagnosis passed), otherwise the stage cannot be entered.

5. Implementation of the new seven QC methods in EAM

  1. Relationship Diagram Method (RelationsDiagram):
    • Application: Clarify the causal propagation network between EAM internal events (such as EAM_AURA_STATE_CHANGED) and WoW events (such as UNIT_AURA) to prevent event storms and infinite returns.
  2. Systematic Chart:
    • Application: Layer large refactoring goals (such as "12.1.0 zero allocation") into sub-module changes (AuraService pooling, CooldownService pooling, AlertManager duplication, renderer multi-type rendering).
  3. Matrix Diagram:
    • Application: The Docs/21_RACI_EXPERTS_MATRIX.md that has been implemented in this case is used to clarify the responsibilities of experts and development tasks.
  4. PDPC method (Process Decision Program Chart/Process Decision Program Chart):
  5. Application: Before writing the program code, predict all possible abnormal paths (such as: the official API returns nil, the data is not cached, and limits are set during combat), and prepare backup countermeasures such as pcall fault isolation, safeCheckIsZero and Util.isSecretValue in advance in the program code.
  6. Arrow Bar Chart Method (Arrow Chart):
  7. Application: Clarify the work order of sub-agent artifacts, avoid multiple agents from writing the same file at the same time, and ensure smooth merging of version branches.
