EventAlertMod 品質管制與要因分析指引 (QC & Root Cause Analysis Guide)

少年欸！為確保 EAM 在重構與日後開發中的程式碼品質，避免在同一個 Bug 上反覆栽跟頭，本文件正式將企業管理學的品質管制（TQM）、新 QC 七大手法、5 Whys（WHY-WHY 分析）、魚骨圖要因分析、對策矩陣評估與甘特圖時程相依精神，全面導入 EAM 的開發與 Bug 診斷流程中。

後續所有 subagent 進行問題排查、方案設計（Implementation Plan）與 PR 審查，均須遵循此 QC 指引。

1. 5 Whys (WHY-WHY 要因分析) 實施規範

當 EAM 出現任何 Bug、Taint 阻斷或 Lua 報錯時，主代理與 subagent 不得直接嘗試修改程式碼，必須先在 Docs/15_DEVELOPMENT_ISSUE_LOG.md 紀錄中進行 5 Whys 分析，連續追問 5 次「為什麼」，直到找出根本原因（Root Cause）。

範例：戰鬥中 IsZero() 布林判定崩潰分析

  • Why 1 (一問)：為什麼 OnUpdate 執行時插件崩潰並阻斷 UI？
    • 答：因為在 Renderer.lua:97 對 durationObj:IsZero() 的傳回值進行了直接布林判斷（if durationObj:IsZero() then）。
  • Why 2 (二問)：為什麼 IsZero() 會引發崩潰？
    • 答：因為該 DurationObject 代表的光環在戰鬥中受到 Secrecy 限制，其 IsZero() 方法返回的是一個受保護的 Secret Boolean。
  • Why 3 (三問)：為什麼受保護的 Secret Boolean 不能做布林判斷？
    • 答：因為魔獸世界 12.x 引擎為了防止 AddOn 藉由 API 判定來輔助戰鬥決策，嚴禁在 Lua 中對 Secret Boolean 執行布林跳轉指令（Boolean Test）。
  • Why 4 (四問)：為什麼我們非得要在 OnUpdate 中呼叫 IsZero()？
    • 答：因為戰鬥中 UNIT_AURA 事件會被官方節流，且 Secret 狀態下我們無法取得剩餘時間以預排 Scheduler，必須在 Renderer 輪詢該 Predicate 以便在光環消失時主動回收 Icon。
  • Why 5 (五問/根本原因)：為什麼之前沒有防護？
    • 答：因為 Renderer 缺乏對 Secret Value 的前置檢測與安全包裹，且不知道 IsZero() 的傳回值也會是 Secret。
  • 根本對策：引入 safeCheckIsZero，利用 Util.isSecretValue 過濾返回值，若為 Secret 則絕不對其進行 Boolean Test。

2. 魚骨圖 (Ishikawa Diagram) 要因分析模板

在遭遇複雜系統性問題時（例如「一個 ICON 都沒出現」或「高頻戰鬥中 UI 嚴重卡頓」），須使用魚骨圖從以下五個維度進行分析：

  【人 (Agent)】                   【機 (Client/WoW API)】         【料 (SavedVariables/Data)】
  - 載入順序快取 nil 漏配           - 12.x Secrecy 加密與限制       - 舊 SavedVariables 資料遷移失效
  - 靜態變數被提早快取             - 事件節流與 CLEU 事件停用       - 預設 Alerts 表為空 (WTF遺失)
          \                             \                               \
           \                             \                               \
            --------------------------------------------------------------------> [ 系統異常/Bug ]
           /                             /                               /
          /                             /                               /
  - 戰鬥中 layout 頻繁重排         - 戰鬥中 InCombatLockdown 限制
  - 輪詢 pcall 產生 Heap 垃圾       - 脫戰後 defer 釋放機制
  【法 (Algorithm/邏輯)】          【環 (Environment/戰鬥狀態)】
  • 人 (Agent / Developer)：AI 代理的代碼載入順序、快取邏輯、靜態變數使用不當。
  • 機 (Client / WoW API)：魔獸客戶端 API 的行為變更、加密限制、事件節流。
  • 料 (SavedVariables / Data)：資料庫 Schema、Defaults 設定檔、本地化字串。
  • 法 (Algorithm / 邏輯流程)：演算法開銷、迴圈設計、事件訂閱、GC 垃圾產生率。
  • 環 (Environment / 遊戲狀態)：戰鬥中/非戰鬥中、低 FPS、大數量 Buff 特殊情境。

3. 對策擬定與評估矩陣 (Countermeasure Matrix)

在 implementation_plan.md 中設計複雜方案時，必須提出至少 兩個候選對策，並以矩陣圖形式進行量化打分評估（1~5 分，5 分為最優），最終選取綜合得分最高者。

範例：解決戰鬥中 Secret 倒數問題的對策評估

對策方案 效果 (Effectiveness) 可行性 (Feasibility) 效能開銷 (Low GC/CPU) 安全風險 (Low Taint/Secret) 綜合得分 決策
對策 A：強行利用 Tooltip Scraping 在每幀 OnUpdate 爬取時間 2 4 1 (頻繁字串 GC) 2 (極易 tainted) 9 淘汰
對策 B：使用 Shadow Host 寄生吸附官方 CooldownViewer 5 3 5 (100% 避讓佈局) 4 (安全隔離) 17 採納 (備用)

4. 甘特圖排程與相依性精神 (Gantt Spirit)

在 task.md 中，必須體現甘特圖的相依性（Dependencies）與關鍵路徑（Critical Path），以防止並行開發的 subagent 發生「因前置代碼未就緒而空轉」或「高耦合代碼衝突」。

排程原則：

  1. Dependency Guard (相依性守衛)：在 task.md 中明確標註前置依賴。例如：
    • [ ] 任務 C (AuraService 重構) [依賴：任務 A (EventRouter 改版)]
  2. Critical Path First (關鍵路徑優先)：將決定架構穩定性（如 Core, SavedVariables）的任務排在最前，分配最昂貴的資源與 A 級審查專家。
  3. Milestone Gate (里程碑關卡)：每個階段結束時，必須設定 Milestone 進行品質查核（如：luac -p 檢查，/eam doctor 診斷通過），否則不得進入下一階段。

5. 新 QC 七大手法在 EAM 的落地

  1. 關係圖法 (Relations Diagram)：
    • 應用：理清 EAM 內部事件（如 EAM_AURA_STATE_CHANGED）與原生 WoW 事件（如 UNIT_AURA）之間的因果傳播網絡，防止事件風暴與無限遞迴。
  2. 系統圖法 (Systematic Diagram)：
    • 應用：將宏大的重構目標（如 "12.1.0 零分配"）層層分解為子模組變更（AuraService 池化、CooldownService 池化、AlertManager 回收、Renderer 多型渲染）。
  3. 矩陣圖法 (Matrix Diagram)：
    • 應用：即本案已落地的 Docs/21_RACI_EXPERTS_MATRIX.md，用來理清專家與開發任務的責任對照。
  4. PDPC 法 (Process Decision Program Chart / 過程決定計劃圖)：
    • 應用：在代碼編寫前，預測所有可能的異常路徑（如：官方 API 返回 nil、資料未快取、戰鬥中 restricted 限制），並在程式碼中預先布置 pcall 故障隔離、safeCheckIsZero 與 Util.isSecretValue 等 fallback 備援對策。
  5. 箭條圖法 (Arrow Diagram)：
    • 應用：理清 subagent 的並行工作順序，避免多個代理同時寫入同一個檔案，確保版本分支的平滑合併。

EventAlertMod Quality Control & Root Cause Analysis Guide (QC & Root Cause Analysis Guide)

Boy! In order to ensure the code quality of EAM during reconstruction and future development, and to avoid repeated failures on the same bug, this document officially introduces the spirit of enterprise management quality control (TQM), the seven new QC techniques, 5 Whys (WHY-WHY analysis), fishbone diagram factor analysis, strategy matrix evaluation, and Gantt chart schedule dependence into the EAM development and bug diagnosis process. All subsequent subagents must follow this QC guideline when conducting problem troubleshooting, implementation plan (Implementation Plan), and PR review.

1. 5 Whys (WHY-WHY factor analysis) implementation specifications

When any bug, Taint blocking or Lua error occurs in EAM, the main agent and subagent are not allowed to directly try to modify the program code. They must first perform a 5 Whys analysis in the Docs/15_DEVELOPMENT_ISSUE_LOG.md record and ask "why" five times in a row until the root cause (Root Cause) is found.

Example: Analysis of IsZero() Blinn Determination Collapse in Combat

  • Why 1 (one question): Why does the plug-in crash and block the UI when OnUpdate is executed?
    • Answer: Because in Renderer.lua:97 a direct Boolean judgment is made on the return value of durationObj:IsZero() (if durationObj:IsZero() then).
  • Why 2 (second question): Why does IsZero() cause a crash?
  • Answer: Because the aura represented by this DurationObject is restricted by Secrecy in combat, its IsZero() method returns a protected Secret Boolean.
  • Why 3 (three questions): Why can’t the protected Secret Boolean make Boolean judgment?
    • Answer: Because the World of Warcraft 12.x engine prevents AddOn from assisting combat decisions through API determination, it is strictly prohibited to execute Boolean jump instructions (Boolean Test) on Secret Boolean in Lua.
  • Why 4 (Four Questions): Why do we have to call IsZero() in OnUpdate?
    • Answer: Because the UNIT_AURA event in battle will be officially throttled, and we cannot obtain the remaining time to pre-schedule the Scheduler in the Secret state, the Predicate must be polled in the Renderer to actively recycle the Icon when the aura disappears.
  • Why 5 (Five questions/root reasons): Why was there no protection before?
  • Answer: Because Renderer lacks pre-detection and security packaging for Secret Value, and does not know that the return value of IsZero() will also be Secret.
  • Basic Countermeasure: Introduce safeCheckIsZero, use Util.isSecretValue to filter the return value, and never perform Boolean Test on it if it is Secret.

2. Fishbone Diagram (Ishikawa Diagram) factor analysis template

When encountering complex systemic problems (such as "not a single ICON appears" or "the UI is severely stuck during high-frequency battles"), a fishbone diagram must be used to analyze from the following five dimensions:

  【人 (Agent)】                   【機 (Client/WoW API)】         【料 (SavedVariables/Data)】
  - 載入順序快取 nil 漏配           - 12.x Secrecy 加密與限制       - 舊 SavedVariables 資料遷移失效
  - 靜態變數被提早快取             - 事件節流與 CLEU 事件停用       - 預設 Alerts 表為空 (WTF遺失)
          \                             \                               \
           \                             \                               \
            --------------------------------------------------------------------> [ 系統異常/Bug ]
           /                             /                               /
          /                             /                               /
  - 戰鬥中 layout 頻繁重排         - 戰鬥中 InCombatLockdown 限制
  - 輪詢 pcall 產生 Heap 垃圾       - 脫戰後 defer 釋放機制
  【法 (Algorithm/邏輯)】          【環 (Environment/戰鬥狀態)】
  • People (Agent/Developer): The code loading sequence, cache logic, and static variables of the AI ​​agent are improperly used.
  • Machine (Client / WoW API): Behavior changes, encryption restrictions, event throttling for the World of Warcraft client API.
  • Material (SavedVariables / Data): Database Schema, Defaults configuration file, localization string.
  • Method (Algorithm / Logic Flow): Algorithm overhead, loop design, event subscription, GC garbage generation rate.
  • Ring (Environment/Game State): special situation in combat/non-combat, low FPS, large number of Buffs.

3. Countermeasure Matrix

When designing a complex solution in implementation_plan.md, at least two candidate countermeasures must be proposed and quantitatively evaluated in the form of a matrix chart (1~5 points, 5 being the best), and the one with the highest comprehensive score is finally selected.

4. Gantt Chart Scheduling and Dependency Spirit (Gantt Spirit)

In task.md, the Dependencies and Critical Path of the Gantt chart must be reflected to prevent parallel development of subagents from "idling due to unready pre-code" or "highly coupled code conflicts".

Scheduling principles:

  1. Dependency Guard: Clearly mark pre-dependencies in task.md. For example:
    • [ ] Task C (AuraService reconstruction) [Depends on: Task A (EventRouter revision)]
  2. Critical Path First: Prioritize tasks that determine the stability of the architecture (such as Core, SavedVariables), and allocate the most expensive resources and A-level review experts.
  3. Milestone Gate (milestone level): At the end of each stage, Milestone must be set for quality inspection (such as: luac -p inspection, /eam doctor diagnosis passed), otherwise it will not enter the next stage.

5. Implementation of the seven new QC techniques in EAM

  1. Relations Diagram:
    • Application: Clarify the causal propagation network between EAM internal events (such as EAM_AURA_STATE_CHANGED) and native WoW events (such as UNIT_AURA) to prevent event storms and infinite recursion.
  2. Systematic Diagram:
    • Application: Break down ambitious refactoring goals (such as "12.1.0 zero allocation") into sub-module changes (AuraService pooling, CooldownService pooling, AlertManager recycling, Renderer multi-type rendering).
  3. Matrix Diagram:
    • Application: The Docs/21_RACI_EXPERTS_MATRIX.md that has been implemented in this case is used to clarify the responsibilities of experts and development tasks.
  4. PDPC method (Process Decision Program Chart / Process Decision Plan Chart):
  5. Application: Before writing code, predict all possible exception paths (such as official API returning nil, data not cached, restricted restrictions in combat), and pre-arrange pcall fault isolation, safeCheckIsZero and Util.isSecretValue and other fallback backup countermeasures in the code.
  6. Arrow Diagram:
  7. Application: Clarify the parallel work sequence of subagent, avoid multiple agents writing the same file at the same time, and ensure smooth merging of version branches.
