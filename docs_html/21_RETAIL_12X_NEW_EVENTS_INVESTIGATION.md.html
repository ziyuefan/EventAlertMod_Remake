魔獸世界 Retail 12.x / Midnight-era 全新與高頻事件調查報告

少年欸！針對您二次提供與追加的 16 個核心事件，我們完成了魔獸世界 12.x 最新 AddOn API 的調查，以下為各事件的底層原理、Payload 引數及 EAM 的具體整合設計方案。

1. 第一批事件詳細調查與 Payload 說明

📌 Event 1: COMBAT_LOG_MESSAGE

  • 原生作用：Patch 12.0.0 後新引進的事件。由於官方將傳統的 COMBAT_LOG_EVENT_UNFILTERED (CLEU) 移出 AddOn 存取權限，且將 CombatLogGetCurrentEventInfo() 設為 Restricted（被污染的插件代碼無法讀取），官方提供了此事件作為 floating combat text (FCT) 浮動文字顯示的格式化替代方案。
  • 引數 Payload：
    • Arg 1 (messageText): string - 已格式化好且在地化的戰鬥文字字串（例如："你的點燃傷害順劈斬訓練目標262點火焰。"）。
    • Arg 2 (r): number - 文字顯示的紅色色值通道。
    • Arg 3 (g): number - 文字顯示的綠色色值通道。
    • Arg 4 (b): number - 文字顯示的藍色色值通道。
    • Arg 5 (displayType): string/number - 訊息的類型或樣式。
  • EAM 整合策略：
    • 限制：此事件不包含 spellID、sourceGUID、destGUID 等結構化數據，只有純文字與顏色，對其做 regex scraping 的 CPU 負擔極高且不具備版本健壯性。
    • 應用：EAM 將其註冊至 EventRouter 作為 debug/trace 日誌，在使用者開啟除錯模式時直接輸出為可讀性極佳的戰鬥事件流，不參與核心的 Aura/Cooldown 計時事實。

📌 Event 2: COMBAT_TEXT_UPDATE

  • 原生作用：魔獸世界傳統的浮動戰鬥文字（Floating Combat Text）更新事件。當被監控的單位（預設需用 CombatTextSetActiveUnit 設定，通常是 "player", "target", "pet"）發生戰鬥狀態更新時觸發。
  • 引數 Payload：
    • Arg 1 (combatTextType): string - 戰鬥文字類型（例如："SPELL_AURA_START", "DAMAGE", "HEAL", "ENERGY", "RAGE", "DODGE" 等）。
  • EAM 整合策略：
    • 主要用於 EventRouter debug trace 監控，在 AuraService 因安全限制突然失效時，可作為獲取 "SPELL_AURA_START" 狀態的防禦性輔助日誌。

📌 Event 3: UNIT_SPELLCAST_SENT

  • 原生作用：當被監控的單位發送施法請求給伺服器時觸發（此時法術尚未真正施放成功，還在 GCD 判定或唱法開始階段）。
  • 引數 Payload：
    • Arg 1 (unitTarget): string - 施法單位（如 "player", "party1"）。
    • Arg 2 (targetName): string - 目標名稱。
    • Arg 3 (castGUID): string - 該次施法的唯一 GUID。
    • Arg 4 (spellID): number - 法術 ID。
  • EAM 整合策略：
    • 可用於 EAM 的施法預響應機制。在唱法（Cast）或瞬發法術發送的瞬間，若對應 spellID 在監控名單中，可預先建立 AlertState 並渲染，提升視覺回饋速度。如果後續收到 UNIT_SPELLCAST_FAILED_QUIET 或 UNIT_SPELLCAST_FAILED，則即時清理，避免 ghost 圖示殘留。

📌 Event 4: UNIT_SPELLCAST_SUCCEEDED

  • 原生作用：當被監控的單位成功施放法術時觸發。
  • 引數 Payload：
    • Arg 1 (unitTarget): string - 施法單位。
    • Arg 2 (castGUID): string - 施法唯一 GUID。
    • Arg 3 (spellID): number - 法術 ID。
  • EAM 整合策略：
    • 核心重構替代方案：這是 12.x 下 GroundEffectService (地面效果監控) 的救星級 API！由於 CLEU 被封閉，我們直接將 GroundEffectService 修改為註冊 UNIT_SPELLCAST_SUCCEEDED。
    • 當 unitTarget == "player" 且 spellID 為我們監控的地面技能（如暴風雪、寒冰寶珠）時，立刻觸發該技能的持續時間倒數。此 API 100% 避開了 CLEU，且 100% 為 Retail 安全 API，完全無 Taint 被隔離的風險。

📌 Event 5: UNIT_SPELLCAST_FAILED_QUIET

  • 原生作用：當單位的施法失敗且不觸發標準的 UI 錯誤語音（例如 GCD 衝突被伺服器拒絕）時觸發。
  • 引數 Payload：
    • Arg 1 (unitTarget): string - 施法單位。
    • Arg 2 (castGUID): string - 該次施法 GUID。
    • Arg 3 (spellID): number - 法術 ID。
  • EAM 整合策略：
    • 用於防禦性狀態重置。當捕獲到玩家自身的此事件時，若該 spellID 的預備狀態或唱法計時存在，第一時間將其回收釋放，確保 UI 的狀態精確同步。

📌 Event 6: CURRENT_SPELL_CAST_CHANGED

  • 原生作用：玩家當前的施法（唱法/引導）狀態發生改變時觸發。
  • 引數 Payload：
    • Arg 1 (cancelledCast): boolean - 前一次施法是否被取消。
  • EAM 整合策略：
    • 用於 EAM 的 Cast/Channeling 輔助監控。如果 cancelledCast == true，代表施法被打斷或因移動中斷，EAM 能即時將關聯的 Alert 圖示隱藏。

📌 Event 7: SPELL_ACTIVATION_OVERLAY_HIDE

  • 原生作用：當原生 UI 螢幕中央的半透明「法術 proc 貼圖提示（C_SpellActivationOverlay）」消失時觸發。
  • 引數 Payload：
    • Arg 1 (spellID): number - 螢幕提示消失的法術 ID。
  • EAM 整合策略：
    • 用於監控大型螢幕貼圖 proc 的消失。這與快捷列的發光不同，但它代表該法術在中央的 proc 動畫已經結束。

📌 Event 8: UNIT_POWER_FREQUENT

  • 原生作用：被監控單位的能量值（聖能、連擊點、真氣等）發生變更時的高頻觸發版本。傳統的 UNIT_POWER_UPDATE 有官方節流，而此事件在急速變動時能保證即時觸發。
  • 引數 Payload：
    • Arg 1 (unitTarget): string - 能量變更單位。
    • Arg 2 (powerTypeToken): string - 能量類型（如 "COMBO_POINTS", "HOLY_POWER"）。
  • EAM 整合策略：
    • 無延遲能量更新：將其引入 ClassPowerService！除註冊 UNIT_POWER_UPDATE 外，同時註冊 UNIT_POWER_FREQUENT，使連擊點、聖能、真氣等資源變動的視覺反應速度達到 0ms 延遲，大幅優化操作流暢度。

2. 第二批事件詳細調查與 Payload 說明

📌 Event 9: SPELL_UPDATE_COOLDOWN

  • 原生作用：當任一技能進入冷卻、冷卻狀態變更或冷卻完成時觸發。
  • 引數 Payload：無引數。
  • EAM 整合策略：
    • 此事件為 CooldownService 的核心事件。每當觸發時，CooldownService 會對監控的冷卻技能列表進行 O(1) 的反應式重新掃描，確保計時圖示精確顯現。

📌 Event 10: SPELL_UPDATE_USABLE

  • 原生作用：當法術的可用性（Usable）狀態改變時觸發（如能量值足夠/不足以施放、或是技能施放條件被滿足/取消）。
  • 引數 Payload：無引數。
  • EAM 整合策略：
    • 可做為 CooldownService 輔助檢測可用性的時機。在除錯模式下，記錄此事件發送，幫助排查部分冷卻技能因資源不足在 CD 轉完後沒有顯示為可用的 bug。

📌 Event 11: SPELL_UPDATE_USES

  • 原生作用：魔獸世界 12.x 新引入事件。當某法術的「可用次數/使用次數（Uses）」或「充能次數（Charges）」改變時觸發。
  • 引數 Payload：
    • Arg 1 (spellID): number - 法術 ID。
    • Arg 2 (uses/charges): number - 剩餘使用次數。
  • EAM 整合策略：
    • 可用於技能充能次數或可用次數改變時，即時更新 CooldownService 中對應技能的 charges 數值，不需要等待整體的 Full Scan，提高反應效率。

📌 Event 12: SPELL_ACTIVATION_OVERLAY_GLOW_HIDE

  • 原生作用：這是一個極其關鍵的事件！ 與 SPELL_ACTIVATION_OVERLAY_HIDE（螢幕中央月牙貼圖消失）不同，這是指快捷列技能圖示金色發光亮框（Action Bar Glow） 消失的事件。其對應的事件是 SPELL_ACTIVATION_OVERLAY_GLOW_SHOW。
  • 引數 Payload：
    • Arg 1 (spellID): number - 金色發光亮框消失的法術 ID。
  • EAM 整合策略：
    • 完美的圖示亮框同步：我們應該監聽 SPELL_ACTIVATION_OVERLAY_GLOW_SHOW 與 SPELL_ACTIVATION_OVERLAY_GLOW_HIDE。
    • 當玩家觸發 Proc（如火法的大火球、冰法的冰指亮框）時，官方會觸發 SHOW，如果 EAM 目前顯示了該技能圖示，我們在其上顯示 Glow 亮框；收到 HIDE 時隱藏發光。這能做到與官方按鈕發光 100% 同步！

3. 第三批追加事件詳細調查與 Payload 說明

📌 Event 13: BAG_UPDATE_COOLDOWN

  • 原生作用：當背包中任一物品（如爐石、使用型飾品、藥水等）進入冷卻、冷卻狀態變更或冷卻結束時觸發。
  • 引數 Payload：無引數。
  • EAM 整合策略：
    • 此事件為 ItemCooldownService 監控物品冷卻的核心事件。觸發時會重新掃描當前被監控的物品冷卻資訊並更新至 UI，避免了 OnUpdate 輪詢背包冷卻的 CPU 開銷。

📌 Event 14: COOLDOWN_VIEWER_SPELL_OVERRIDE_UPDATED

  • 原生作用：魔獸 12.x 原生 CooldownViewer (冷卻管理器/CDM) 專用內部事件。當法術覆蓋關係（Spell Override，即因為天賦、形態或某些特殊狀態，使一個法術被另一個法術替換，例如薩滿冰霜震擊變成冰怒）發生變更時由客戶端觸發。
  • 引數 Payload：
    • Arg 1 (overriddenSpellID): number - 被覆蓋/新替換的法術 ID。
    • Arg 2 (originalSpellID): number - 原始法術 ID。
  • EAM 整合策略：
    • 解決技能 Override 的計時 Bug：對 EAM 的冷卻監控非常有用！以往玩家點了天賦使法術 ID 變更時，EAM 無法第一時間動態跟隨。我們將在 CooldownService 監聽此事件，當偵測到被監控的法術被 Override 時，動態將對應狀態的 spellID 或是監控參數更新為 override 後的新 ID，完美避免冷卻失效。

📌 Event 15: ASSISTED_COMBAT_ACTION_SPELL_CAST

  • 原生作用：魔獸 12.x 針對「輔助功能（Assisted Combat / 輔助點擊 / Hold to Cast 連續施法）」新引進的輔助施法觸發事件。當玩家使用輔助系統進行法術施放時觸發。
  • 引數 Payload：通常攜帶 spellID 及 castGUID 等。
  • EAM 整合策略：
    • 這是一個全新的內部輔助施法事件。EAM 將其註冊至 EventRouter 作為除錯追蹤。當玩家開啟了原生輔助施法且觸發該事件時，可用於輔助 UNIT_SPELLCAST_SUCCEEDED 的雙重判定，防範極限高頻施法時漏判。

📌 Event 16: ACTIONBAR_UPDATE_STATE

  • 原生作用：快捷列狀態更新事件。當快捷列上的任何技能、物品的狀態（例如變灰、變亮、可用性、冷卻等）改變時觸發。
  • 引數 Payload：無引數。
  • EAM 整合策略：
    • 主要做為 Debug Trace 記錄，以及在特殊狀態下（如切換動作條頁面）觸發 CooldownService 和 ItemCooldownService 做防禦性一次性刷新，確保圖示視覺一致。

World of Warcraft Retail 12.x / Midnight-era New and High Frequency Incident Investigation Report

Boy! Regarding the 16 core events you provided and added for the second time, we have completed the investigation of the latest AddOn API of World of Warcraft 12.x. The following is the underlying principle of each event, the payload parameters and the specific integration design plan of EAM.

1. Detailed investigation and payload description of the first batch of incidents

📌Event 1: COMBAT_LOG_MESSAGE

  • Native function: Newly introduced events after Patch 12.0.0. Since the official moved the traditional COMBAT_LOG_EVENT_UNFILTERED (CLEU) out of AddOn access rights and set CombatLogGetCurrentEventInfo() to Restricted (unreadable by tainted plug-in code), the official provided this event as a formatting alternative for floating combat text (FCT) floating text display.
  • Argument Payload:
  • Arg 1 (messageText): string - Formatted and localized combat text string (eg: "Your Ignite damage Cleave training target is 262 points of Fire.").
    • Arg 2 (r): number - The red color channel of the text display.
    • Arg 3 (g): number - The green color channel of the text display.
    • Arg 4 (b): number - The blue color channel of the text display.
    • Arg 5 (displayType): string/number - The type or style of the message.
  • EAM Integration Strategy:
    • Limitations: This event does not contain structured data such as spellID, sourceGUID, destGUID, etc., only plain text and color. The CPU load of regex scraping on it is extremely high and it does not have version robustness.
    • Application: EAM registers it to EventRouter as a debug/trace log, and directly outputs it as a highly readable combat event stream when the user turns on the debug mode, without participating in the core Aura/Cooldown timing facts.

📌 Event 2: COMBAT_TEXT_UPDATE

  • Native function: The traditional Floating Combat Text update event in World of Warcraft. Triggered when the combat status of the monitored unit (default needs to be set with CombatTextSetActiveUnit, usually "player", "target", "pet") occurs.
  • Argument Payload:
    • Arg 1 (combatTextType): string - Combat text type (for example: "SPELL_AURA_START", "DAMAGE", "HEAL", "ENERGY", "RAGE", "DODGE", etc.).
  • EAM Integration Strategy:
    • Mainly used for EventRouter debug trace monitoring. When AuraService suddenly fails due to security restrictions, it can be used as a defensive auxiliary log to obtain the "SPELL_AURA_START" status.

📌Event 3: UNIT_SPELLCAST_SENT

  • Native function: Triggered when the monitored unit sends a spell casting request to the server (at this time, the spell has not actually been cast successfully, and is still in the GCD judgment or singing start stage).
  • Argument Payload:
  • Arg 1 (unitTarget): string - The casting unit (e.g. "player", "party1").
    • Arg 2 (targetName): string - target name.
    • Arg 3 (castGUID): string - The unique GUID for this cast.
    • Arg 4 (spellID): number - The spell ID.
  • EAM Integration Strategy:
  • Casting Pre-Response Mechanism available for EAM. At the moment when a cast or instant spell is sent, if the corresponding spellID is in the monitoring list, the AlertState can be pre-created and rendered to increase the speed of visual feedback. If you subsequently receive UNIT_SPELLCAST_FAILED_QUIET or UNIT_SPELLCAST_FAILED, clean them immediately to avoid ghost icons remaining.

📌 Event 4: UNIT_SPELLCAST_SUCCEEDED

  • Native Effect: Triggered when the monitored unit successfully casts a spell.
  • Argument Payload:
    • Arg 1 (unitTarget): string - The casting unit.
    • Arg 2 (castGUID): string - The cast unique GUID.
    • Arg 3 (spellID): number - The spell ID.
  • EAM Integration Strategy:
    • Core Refactor Alternative: This is the star-saving API for GroundEffectService (ground effects monitoring) under 12.x! Since CLEU is closed, we directly modify GroundEffectService to register UNIT_SPELLCAST_SUCCEEDED.
  • When unitTarget == "player" and spellID is a ground skill we monitor (such as Blizzard, Ice Orb), the duration countdown of the skill will be triggered immediately. This API 100% avoids CLEU and is 100% Retail safe API, with no risk of Taint being quarantined.

📌 Event 5: UNIT_SPELLCAST_FAILED_QUIET

  • Native Effect: Triggered when a unit's spellcasting fails and does not trigger the standard UI error voice (e.g. GCD conflict rejected by server).
  • Argument Payload:
    • Arg 1 (unitTarget): string - The casting unit.
    • Arg 2 (castGUID): string - The cast GUID.
    • Arg 3 (spellID): number - The spell ID.
  • EAM Integration Strategy:
    • Used for defensive status reset. When the player's own event is captured, if the spellID's preparation state or singing timing exists, it will be recycled and released as soon as possible to ensure accurate synchronization of the UI state.

📌 Event 6: CURRENT_SPELL_CAST_CHANGED

  • Native function: Triggered when the player's current casting (singing/guiding) status changes.
  • Argument Payload:
    • Arg 1 (cancelledCast): boolean - Whether the previous cast was canceled.
  • EAM Integration Strategy:
    • Cast/Channeling auxiliary monitoring for EAM. If cancelledCast == true, it means that the casting was interrupted or interrupted by movement, and EAM can hide the associated Alert icon immediately.

📌 Event 7: SPELL_ACTIVATION_OVERLAY_HIDE

  • Native function: Triggered when the translucent "Spell proc texture prompt (C_SpellActivationOverlay)" in the center of the native UI screen disappears.
  • Argument Payload:
    • Arg 1 (spellID): number - The spell ID for which the screen prompt disappears.
  • EAM Integration Strategy:
    • Used to monitor the disappearance of large screen map procs. This is different from the hotbar glow, but it indicates that the spell's central proc animation has ended.

📌 Event 8: UNIT_POWER_FREQUENT

  • Native function: A high-frequency trigger version when the energy value of the monitored unit (holy energy, combo points, infuriating energy, etc.) changes. The traditional UNIT_POWER_UPDATE has official throttling, and this event is guaranteed to be triggered immediately when there are rapid changes.
  • Argument Payload:
    • Arg 1 (unitTarget): string - Energy change unit.
    • Arg 2 (powerTypeToken): string - The energy type (e.g. "COMBO_POINTS", "HOLY_POWER").
  • EAM Integration Strategy:
  • Delay-free energy updates: Introduce it into ClassPowerService! In addition to registering UNIT_POWER_UPDATE, register UNIT_POWER_FREQUENT at the same time, so that the visual response speed of resource changes such as combo points, holy energy, and infuriating energy reaches 0ms delay, greatly optimizing the smoothness of the operation.

2. Detailed investigation and payload description of the second batch of incidents

📌 Event 9: SPELL_UPDATE_COOLDOWN

  • Native function: Triggered when any skill enters cooldown, cooldown status changes, or cooldown is completed.
  • Parameters Payload: No arguments.
  • EAM Integration Strategy:
    • This event is the core event of CooldownService. Whenever triggered, CooldownService will perform an O(1) reactive rescan on the monitored cooling skill list to ensure that the timing icon is displayed accurately.

📌 Event 10: SPELL_UPDATE_USABLE

  • Native function: Triggered when the spell's availability (Usable) status changes (such as the energy value is sufficient/insufficient to cast, or the skill casting conditions are met/cancelled).
  • Parameters Payload: No arguments.
  • EAM Integration Strategy:
  • Can be used as an opportunity for CooldownService to assist in checking availability. In debugging mode, record this event to help troubleshoot the bug that some cooling skills are not displayed as available after CD transfer due to insufficient resources.

📌 Event 11: SPELL_UPDATE_USES

  • Native Function: Newly introduced events in World of Warcraft 12.x. Triggered when the "Uses" or "Charges" of a spell changes.
  • Argument Payload:
  • Arg 1 (spellID): number - The spell ID.
    • Arg 2 (uses/charges): number - Number of remaining uses.
  • EAM Integration Strategy:
    • When the number of charges or available times for a skill changes, the charges value of the corresponding skill in CooldownService is updated immediately, without waiting for the overall Full Scan, improving response efficiency.

📌 Event 12: SPELL_ACTIVATION_OVERLAY_GLOW_HIDE

  • Native Effect: This is an extremely critical event! ** Different from SPELL_ACTIVATION_OVERLAY_HIDE (the crescent moon texture in the center of the screen disappears), this refers to the event where the shortcut bar skill icon golden glowing bright frame (Action Bar Glow)** disappears. The corresponding event is SPELL_ACTIVATION_OVERLAY_GLOW_SHOW.
  • Argument Payload:
    • Arg 1 (spellID): number - The spell ID that causes the golden glowing frame to disappear.
  • EAM Integration Strategy:
  • Perfect icon highlight synchronization: We should listen for SPELL_ACTIVATION_OVERLAY_GLOW_SHOW and SPELL_ACTIVATION_OVERLAY_GLOW_HIDE.
    • When the player triggers Proc (such as fire spell's big fire ball, ice spell's ice finger bright frame), the official will trigger SHOW. If EAM currently displays the skill icon, we will display the Glow bright frame on it; when receiving HIDE, the glow will be hidden. This is 100% synchronized with the official button glow!

3. Detailed investigation and payload description of the third batch of additional events

📌 Event 13: BAG_UPDATE_COOLDOWN

  • Native function: Triggered when any item in the backpack (such as hearthstone, usable accessories, potions, etc.) enters cooling, changes the cooling state, or ends cooling.
  • Parameters Payload: No arguments.
  • EAM Integration Strategy:
  • This event is the core event of ItemCooldownService monitoring item cooling. When triggered, the currently monitored item cooling information will be rescanned and updated to the UI, avoiding the CPU overhead of OnUpdate polling for backpack cooling.

📌 Event 14: COOLDOWN_VIEWER_SPELL_OVERRIDE_UPDATED

  • Native function: Warcraft 12.x native CooldownViewer (cooling manager/CDM) dedicated internal event. Triggered by the client when the spell override relationship (Spell Override, that is, one spell is replaced by another spell due to talent, form or some special status, such as Shaman Frost Shock becomes Ice Fury) changes.
  • Argument Payload:
    • Arg 1 (overriddenSpellID): number - The overridden/newly replaced spell ID.
  • Arg 2 (originalSpellID): number - The original spell ID.
  • EAM Integration Strategy:
  • Fixed skill Override timing bug: Cooldown monitoring for EAM is very useful! In the past, when a player clicked a talent to change the spell ID, EAM could not dynamically follow it immediately. We will listen to this event in CooldownService. When it is detected that the monitored spell is overridden, the spellID or monitoring parameter of the corresponding state will be dynamically updated to the new ID after override, perfectly avoiding cooling failure.

📌 Event 15: ASSISTED_COMBAT_ACTION_SPELL_CAST

  • Native function: Warcraft 12.x newly introduced assisted casting trigger event for "Assisted Combat / Assisted Click / Hold to Cast Continuous Casting". Triggered when the player uses the auxiliary system to cast a spell.
  • Argument Payload: usually carries spellID and castGUID, etc.
  • EAM Integration Strategy:
  • This is a new internal assisted spellcasting event. EAM registers it with EventRouter as a debug trace. When the player turns on the native auxiliary casting and triggers this event, it can be used to assist the double judgment of UNIT_SPELLCAST_SUCCEEDED to prevent missed judgments during extreme high-frequency casting.

📌 Event 16: ACTIONBAR_UPDATE_STATE

  • Native function: Quick column status update event. Triggered when the status of any skill or item on the shortcut bar changes (such as graying out, brightening, availability, cooldown, etc.).
  • Parameters Payload: No arguments.
  • EAM Integration Strategy:
  • Mainly used as Debug Trace records, and in special states (such as switching action bar pages) to trigger CooldownService and ItemCooldownService for defensive one-time refresh to ensure visual consistency of the icons.
