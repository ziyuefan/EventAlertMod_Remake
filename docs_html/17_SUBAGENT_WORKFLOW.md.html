Subagent 工作流規劃

本文件定義 EventAlertMod 開發時何時規劃並使用 subagent。目標是讓大型工作可並行、可追溯、可控風險；不是把所有任務都外包，也不是增加不必要的協調成本。

基本原則

  • 只有在使用者已授權 subagent、且任務可拆成清楚邊界時才使用。
  • 主代理必須先判斷 critical path（關鍵路徑）：下一步立即阻塞的工作由主代理自己處理。
  • Subagent 適合處理 sidecar task（旁路任務）：可並行、具體、可驗收、不阻塞主代理下一步。
  • 不重複派工；同一問題已有 subagent 結果時，除非有新證據或不同角度，不再派第二個做同樣分析。
  • 程式修改型 subagent 必須有明確寫入範圍，避免多個代理改同一批檔案。
  • 所有 subagent 結果都需由主代理整合與最終判斷，不直接視為已驗證事實。

適合使用 subagent 的情境

  • 大型 rewrite pass 中，可切成互不重疊模組：
  • Services/AuraService.lua
  • Services/CooldownService.lua
  • UI/Renderer.lua
  • Debug/PromptExport.lua
  • API 查證與文件整理可和本地實作並行，例如：
  • 12.x API change 來源複核。
  • Secret Values / taint 討論整理。
  • DurationObject / DurationTextBinding 實作方式比較。
  • 驗證與掃描可與修補並行，例如：
  • 搜尋舊 API 直接呼叫。
  • 檢查 table.freeze 誤用。
  • 檢查 C_Timer.After(function() ...) 熱路徑。
  • 對 Docs 與 AGENTS 進行一致性檢查。
  • 大量文件中文化、術語統一、測試清單補齊。
  • Package / CurseForge 發佈流程的旁路檢查，例如排除清單、TOC 版本、zip 命名規則。

不適合使用 subagent 的情境

  • 小型單檔修改，主代理可直接完成。
  • 下一步完全依賴該結果，等待 subagent 反而拖慢 critical path。
  • 需要即時判斷使用者意圖、風險或授權的操作。
  • 涉及刪除、搬移、覆寫大量檔案，而尚未完成備份與範圍確認。
  • 高耦合修改，例如同時改 SavedVariables schema、migration、Options UI、Slash command 且尚未界定寫入範圍。
  • 尚未有明確驗收條件的探索性問題。

角色使用建議

  • explorer：用於具體問題的只讀調查，例如「找出 active Lua 中所有 cooldown API 呼叫」。
  • worker：用於可分割的實作，例如「只修改 UI/Options.lua，新增某個設定欄位，不碰 SavedVariables」。

除非任務有明確理由，subagent 使用預設繼承模型；不主動指定更昂貴或不同模型。

  • 若 subagent 修改程式碼，主代理仍需執行專案靜態驗證。
  • 若 subagent 提供 API 資訊，主代理需標明來源層級：
  • 官方文件或 Warcraft Wiki。
  • 使用者提供。
  • 搜尋索引。
  • 實機驗證。
  • 若 subagent 遇到工具限制、API 不確定性或錯誤，需追加到 Docs/15_DEVELOPMENT_ISSUE_LOG.md。

RACI 專家分工與 PR 審查原則

為了使 16 位 AI 專家在不同開發任務的定位明確，專案實施 RACI (Responsible, Accountable, Consulted, Informed) 分工定位。

詳細 RACI 矩陣配置與原則見：Docs/21_RACI_EXPERTS_MATRIX.md。

後續所有 subagent 派工時需注意： 1. R (Responsible)：派工時，應指定被列為該任務領域 R 的專家作為 subagent 的 Role 與職能（例如修改 Class DB 時指派 EAM_Class_Expert）。 2. A (Accountable)：Subagent 提交 PR 或成果後，主代理必須交給被列為該任務領域 A 的專家進行審查與批准。 3. C (Consulted)：Subagent 在開發中遇到疑難問題時，必須主動向該任務領域被列為 C 的專家諮詢。

專案品質管制與要因分析原則

為確保代碼品質與開發的嚴謹性，本專案導入 5 Whys (WHY-WHY 要因分析)、魚骨圖、對策評估矩陣與 PDPC 異常防禦機制。

詳細 QC 規範與指引見：Docs/22_QC_ROOT_CAUSE_ANALYSIS_GUIDE.md。

後續開發遵循要求： 1. Bug 診斷：遇任何 runtime 崩潰或邏輯 Bug，必須先執行 5 Whys 分析 與 魚骨圖要因分析，追查根本原因，並寫入問題紀錄中。 2. 對策擬定：設計複雜方案時，必須在 implementation_plan.md 提出至少兩個候選方案，並以對策矩陣從效果、可行性、效能、安全等維度進行量化打分。 3. 相依排程：task.md 任務必須標明依賴關係（甘特圖精神），確保並行開發不發生衝突。

Subagent workflow planning

This document defines when to plan and use subagent when developing EventAlertMod. The goal is to make large-scale work parallel, traceable, and controllable risks; not to outsource all tasks, nor to increase unnecessary coordination costs.

Basic principles

  • Only used when the user has authorized the subagent and the task can be broken down into clear boundaries.
  • The main agent must first determine the critical path: the next immediately blocked work will be handled by the main agent itself.
  • Subagent is suitable for processing sidecar tasks: it can be parallelized, specific, acceptable, and does not block the next step of the main agent.
  • No duplicate work will be assigned; when the same problem already has subagent results, a second one will not be assigned to do the same analysis unless there is new evidence or a different perspective.
  • Program modification subagent must have a clear writing range to prevent multiple agents from modifying the same batch of files.
  • All subagent results are subject to integration and final judgment by the main agent and are not directly regarded as verified facts.

Suitable situations for using subagent

  • Large rewrite passes can be divided into non-overlapping modules:
  • Services/AuraService.lua
  • Services/CooldownService.lua
  • UI/Renderer.lua
  • Debug/PromptExport.lua
  • API verification and documentation can be done in parallel with local implementation, for example:
  • 12.x API change source review.
  • Secret Values ​​/ taint discussion and organization.
  • DurationObject / DurationTextBinding implementation comparison.
  • Verification and scanning can be done in parallel with patching, for example:
  • Search for old API direct calls.
  • Check for table.freeze misuse.
  • Check C_Timer.After(function() ...) hot path. -Consistency check between Docs and AGENTS.
  • Unified culture and terminology in a large number of documents, and completed test lists.
  • Bypass checks for Package / CurseForge release process, such as exclusion lists, TOC versions, zip naming rules.

Situations not suitable for using subagent

  • Small single file modifications can be completed directly by the main agent.
  • The next step depends entirely on this result, and waiting for the subagent will slow down the critical path.
  • Operations that require immediate judgment of user intent, risk, or authorization.
  • Involves deleting, moving, and overwriting a large number of files without completing backup and scope confirmation.
  • Highly coupled modifications, such as changing the SavedVariables schema, migration, Options UI, and Slash command at the same time and the writing range has not yet been defined.
  • Exploratory questions for which acceptance conditions are not yet clear.

Suggestions for using roles

  • explorer: Read-only investigation for specific questions, such as "Find all cooldown API calls in active Lua".
  • Worker: used for separable implementations, such as "only modify UI/Options.lua, add a certain setting field, and do not touch SavedVariables". Unless there is a clear reason for the task, subagent uses the default inheritance model; does not actively specify a more expensive or different model.

Master agent integration rules

  • After Subagent is completed, the master agent needs to quickly review the results and file differences.
  • If the subagent modifies the code, the main agent still needs to perform static verification of the project.
  • If the subagent provides API information, the main agent needs to indicate the source level: -Official documentation or Warcraft Wiki.
  • User provided.
  • Search index.
  • Real machine verification.
  • If the subagent encounters tool limitations, API uncertainties or errors, append to Docs/15_DEVELOPMENT_ISSUE_LOG.md.
  • Markdown absolute facts and HTML conversion rules:
  • The absolute guidance document for this project and subagent development collaboration will always use AGENTS.md and the .md file named in its content as the only fact and Facts-of-Truth reference.
  • The HTML version is for human readability and preview purposes only. When Subagent performs development, read-only analysis, and writeback, it must always use the .md original file as the only factual basis, and HTML files cannot be used as the factual reference.
  • If the task modifies the .md file or AGENTS.md under docs, the conversion tool must be executed after the modification and the corresponding .html file under docs_html/ is updated to ensure consistency.

RACI Expert Division of Labor and PR Review Principles

In order to make the positioning of 16 AI experts in different development tasks clear, the project implemented RACI (Responsible, Accountable, Consulted, Informed) division of labor positioning.

For detailed RACI matrix configuration and principles, see: Docs/21_RACI_EXPERTS_MATRIX.md. Please note the following when dispatching all subsequent subagents: 1. R (Responsible): When dispatching workers, the expert listed as R in the task area should be designated as the role and function of the subagent (for example, when modifying Class DB, assign EAM_Class_Expert). 2. A (Accountable): After Subagent submits a PR or achievement, the master agent must submit it to an expert listed as A in the task area for review and approval. 3. C (Consulted): When Subagent encounters difficult problems during development, it must proactively consult experts listed as C in the task field.

Project quality control and factor analysis principles

In order to ensure code quality and development rigor, this project introduced 5 Whys (WHY-WHY factor analysis), fishbone diagram, countermeasure evaluation matrix and PDPC anomaly defense mechanism.

For detailed QC specifications and guidelines, see: Docs/22_QC_ROOT_CAUSE_ANALYSIS_GUIDE.md. Subsequent development follows the requirements: 1. Bug Diagnosis: In case of any runtime crash or logic bug, you must first perform 5 Whys Analysis and Fishbone Diagram Analysis to trace the root cause and write it into the problem record. 2. Countermeasure formulation: When designing a complex solution, at least two candidate solutions must be proposed in implementation_plan.md, and quantitatively scored from dimensions such as effect, feasibility, efficiency, and safety using a strategy matrix. 3. Dependent Scheduling: task.md Tasks must indicate dependencies (Gantt chart spirit) to ensure that parallel development does not conflict.
