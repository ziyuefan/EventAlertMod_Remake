子代理程式工作流程規劃

本文件定義 EventAlertMod 開發時規劃並使用子代理程式。目標是讓大型工作可額外、可延期、可控制風險；不是把任務全部外包，也不是所有不必要的協調成本增加。

基本原則

  • 只有在使用者已授權子代理、且任務可拆除時才能使用明確的邊界。
  • 主代理先判斷關鍵路徑（關鍵路徑）：下一步立即停止的工作必須由主代理自己處理。
  • 子代理程式處理 sidecar 任務（旁路任務）：可完成、具體、可驗收、不停止主代理下一步。
  • 不重複派工；同一問題存在子代理結果時，除非有新證據或不同角度，不再派第二個做同樣分析。
  • 程式修改型子代理程式必須有明確的書寫範圍，避免多個代理程式改同一個檔案。
  • 所有分代理結果均需由主代理整合與最終判斷，不直接視為已驗證事實。

適合使用子代理人的姿勢

  • 大型重寫頻道中，可精密互不重疊模組：
  • Services/AuraService.lua
  • Services/CooldownService.lua
  • UI/Renderer.lua
  • Debug/PromptExport.lua
  • API 查詢與文件整理可和本地實作家具，例如：
  • 12.x API 更改來源複核。
  • 秘密價值/污染討論整理。
  • DurationObject / DurationTextBinding 實踐方式比較。
  • 驗證與掃描可與修復零件，例如：
  • 搜尋舊API直接呼叫。
  • 檢查table.freeze錯誤用。
  • 檢查 C_Timer.After(function() ...) 熱路徑。
  • 對 Docs 與 AGENTS 進行一致性檢查。
  • 大量文件中文化、術語統一、測試清單補齊。
  • Package / CurseForge 發布流程的旁路檢查，例如排除清單、TOC 版本、zip 命名規則。

不適合使用子代理人的姿勢

  • 小型單檔修改，主代理可直接完成。
  • 下一步完全依賴該結果，等待子代理改為拖慢關鍵路徑。
  • 需要立即判斷使用者意圖、風險或授權的操作。
  • 涉及刪除、搬遷、覆寫大量檔案，而尚未備份與範圍確認。
  • 高修改連接，例如同時改變 SavedVariables schema、migration、Options UI、Slash command 且尚未開始開始範圍。
  • 尚無明確的探索性問題。

角色使用建議

  • explorer：用於特定問題的複雜調查，例如「尋找活動 Lua 中所有冷卻時間 API 通話」。 -worker：用於可分割的實作，例如「只修改UI/Options.lua，新增某個設定欄位，不碰SavedVariables」。 除非任務有明確理由，子代理程式使用預設繼承模型；不指定較昂貴或不同模型。

  • 分代理完成後，主代理需快速審查結果與文件差異。
  • 若子代理程式修改方案代碼，主代理程式仍需執行專案靜態驗證。
  • 若子代理提供 API 資訊，主代理人需標明來源系統：

  • 用戶提供。
  • 搜尋索引。
  • 實機驗證。

  • HTML 版本人類好讀與預覽使用。子代理執行開發、唯讀分析與回寫時，必須一律以 .md 原檔為唯一事實基準，不得以 HTML 文件作為事實參考。

RACI 專家分工與PR審查原則

為了讓 16 位 AI 專家在不同開發任務的定位明確，專案實施 RACI (Responsible, Accountable, Consulted, Informed) 分工定位。

詳細RACI矩陣配置與原則見：Docs/21_RACI_EXPERTS_MATRIX.md。 後續所有子代理派工時需注意： 1. R (Responsible)：派工時，應指定被屬於該任務領域 R 的專家作為子代理的角色與職能（例如修改 Class DB 時指派 EAM_Class_Expert）。 2. A (Accountable)：子代理提交 PR 或結果後，主代理必須遷移到該任務領域 A 的專家進行審查與批准。 3. C（諮詢）：子代理在開發中遇到疑難問題時，必須主動向該任務領域被來自 C 的專家諮詢。

專案品質管制與要因分析原則

為確保程式碼品質與開發的嚴謹性，本專案導入5 Whys (WHY-WHY要因分析)、魚骨圖、對策評估矩陣與PDPC異常防禦機制。

詳細QC規格與指引請見：Docs/22_QC_ROOT_CAUSE_ANALYSIS_GUIDE.md。 後續開發遵循要求： 1. Bug診斷：遇到任何運行時崩潰或邏輯Bug，必須先執行5個為什麼分析與魚骨圖要因分析，追查根本原因，並讀取問題記錄中。 2. 對策擬定：設計複雜方案時，必須在 implementation_plan.md 提出至少兩個候選方案，並以對策矩陣從效果、吸力、感覺、安全等維度進行量化評分。 3. 相依排程：task.md 任務必須標示依賴（關係甘特圖精神），確保任務開發不會發生衝突。

Subagent workflow planning

This document defines EventAlertMod for planning and using subagents during development. The goal is to allow large-scale work to be added, postponed, and risks controllable; not to outsource all tasks, nor to increase all unnecessary coordination costs.

Basic principles

  • Explicit boundaries can only be used if the user has authorized the subagent and the task can be removed.
  • The main agent first determines the critical path (critical path): the work that is stopped immediately in the next step must be handled by the main agent itself.
  • The subagent processes sidecar tasks (bypass tasks): can be completed, specific, acceptable, and does not stop the main agent for the next step.
  • No duplicate work will be assigned; when the same problem has sub-agent results, a second agent will not be assigned to do the same analysis unless there is new evidence or a different perspective.
  • Program modification subagent must have a clear writing range to prevent multiple agents from modifying the same file.
  • All sub-agent results are subject to integration and final judgment by the main agent and are not directly regarded as verified facts.

Suitable posture for using subagents

  • In large rewrite channels, modules can be precisely non-overlapping:
  • Services/AuraService.lua
  • Services/CooldownService.lua
  • UI/Renderer.lua
  • Debug/PromptExport.lua
  • API Query and file organization can be implemented with local furniture, for example:
  • 12.x API Change source review.
  • Secret value/contamination discussion organized.
  • DurationObject / DurationTextBinding Comparison of practical methods.
  • Verify and scan parts that can be repaired, such as:
  • Search for old API calls directly.
  • Check for table.freeze errors.
  • Check C_Timer.After(function() ...) hot path.
  • Conformance check between Docs and AGENTS.
  • Unified culture and terminology in a large number of documents, and completed test lists.
  • Package/CurseForge Bypass checks for the release process, such as exclusion lists, TOC versions, zip naming rules.

Unsuitable posture for using subagents

  • Small single file modifications can be completed directly by the main agent.
  • The next step relies entirely on this result, waiting for the subagent to slow down the critical path instead.
  • Operations that require immediate judgment of user intent, risk, or authorization.
  • Involves deleting, relocating, and overwriting a large number of files without backup and scope confirmation.
  • Highly modified connections, such as changing SavedVariables schema, migration, Options UI, Slash command at the same time and the start scope has not yet started.
  • No clear exploratory questions yet.

Suggestions for using roles

  • explorer: Used for complex investigations of specific problems, such as "Find all cooldown API calls in active Lua". -worker: used for separable implementations, such as "只修改UI/Options.lua, add a setting field, do not touch SavedVariables". Unless there is a clear reason for the task, subagents use the default inheritance model; do not specify a more expensive or different model.

Master agent integration rules

  • After the sub-agent is completed, the main agent needs to quickly review the results and document differences.
  • If the subagent modifies the project code, the main agent still needs to perform static verification of the project.
  • If the sub-agent provides API information, the main agent needs to indicate the source system:
  • Official documentation or the Warcraft Wiki.
  • User provided.
  • Search index.
  • Real machine verification.
  • If the subagent encounters tool limitations, API uncertainties, or errors, it needs to be appended to Docs/15_DEVELOPMENT_ISSUE_LOG.md.
  • Markdown absolute facts and HTML conversion code:
  • The absolute guidance document for this project and subagent development collaboration will always use AGENTS.md and the .md file named in its content as the only fact and Facts-of-Truth reference.
  • HTML version for human readability and preview use. When the subagent performs development, read-only analysis, and writeback, it must always use the .md original file as the only factual basis, and must not use the HTML file as the factual reference.
  • If the task modifies the .md file or AGENTS.md under the file, the conversion tool must be executed after the modification and the corresponding .html file under docs_html/ is updated to ensure consistency.

RACI Expert division of labor and PR review principles

In order to make the positioning of 16 AI experts in different development tasks clear, the project implemented RACI (Responsible, Accountable, Consulted, Informed) division of labor positioning.

For detailed RACI matrix configuration and principles, see: Docs/21_RACI_EXPERTS_MATRIX.md. All subsequent sub-agents should pay attention to the following when dispatching work: 1. R (Responsible): When dispatching work, the role and function of the sub-agent should be assigned to the expert who belongs to the task area R (for example, when modifying Class DB, assign E​​AMCODE_5). 2. A (Accountable): After the sub-agent submits a PR or result, the main agent must be moved to A experts in the task area for review and approval. 3. C (Consultation): When the sub-agent encounters difficult problems during development, he must take the initiative to consult experts from C in the task field.

Project quality control and factor analysis principles

In order to ensure the quality of the program code and the rigor of development, this project introduced 5 Whys (WHY-WHY factor analysis), fishbone diagram, countermeasure evaluation matrix and PDPC anomaly defense mechanism.

For detailed QC specifications and guidelines, please see: Docs/22_QC_ROOT_CAUSE_ANALYSIS_GUIDE.md. Subsequent development follows the requirements: 1. Bug Diagnosis: If you encounter any runtime crash or logic bug, you must first perform 5 why analysis and fishbone diagram analysis to trace the root cause and read the problem record. 2. Countermeasure formulation: When designing a complex solution, at least two candidate solutions must be proposed in implementation_plan.md, and quantitatively scored using a countermeasure matrix from dimensions such as effect, suction, feeling, and safety. 3. Dependent Scheduling: task.md Tasks must be marked with dependencies (relationship Gantt chart spirit) to ensure that there will be no conflicts in task development.
